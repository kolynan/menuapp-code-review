### 1.4 Root cause analysis

After `closeSession()` executes (sessionHelpers.js:158-186):
1. `TableSession.update(sessionId, { status: 'closed' })` — SOM doesn't read this
2. `Order.filter({ table_session: sessionId })` → bulk close all orders to `status: 'closed'`
3. `ServiceRequest.filter({ table: tableId })` → mark all open requests as `done`

After refetch (SOM line 4154-4155):
- `refetchOrders()` — reloads all partner orders
- `refetchRequests()` — reloads all partner service requests

**Expected behavior after successful closeSession + refetch:**
- All orders in group have `status: 'closed'`
- `hasActiveOrder`: all orders are at finish stage with status 'closed' → `!config.isFinishStage` is false → **false**
- `hasServedButNotClosed`: all orders at finish stage with `status === 'closed'` → `o.status !== 'closed'` is false → **false**
- `hasActiveRequest`: all requests marked 'done' → filtered out by `activeRequests` memo (line 3548) → **false**
- Result: group moves to "completed" tab ✓

**BUT if ServiceRequest.filter fails (Fix B), `hasActiveRequest` stays true → group remains Active.**

This means Fix A's root cause is actually Fix B (ServiceRequest filter not working). However, there's a secondary issue:

**Race condition / invalidation gap:** Between `closeSession()` completing and `refetchOrders()`/`refetchRequests()` completing, there's a window where stale data shows. The `await closeSession()` is sequential before refetch, so this should be minimal. But polling intervals could cause other SOM instances to show stale data.

### 1.5 Recommended approach

**Option A: Add useQuery for TableSession.filter({ status: 'open' })** → map openSessionByTableId → override in filteredGroups.

- Pro: Authoritative — session status is the source of truth after closeSession
- Pro: Works even if Fix B fails (ServiceRequests not closed)
- Con: Additional B44 query (1 extra request per polling interval)
- Con: Requires adding TableSession import and wiring through components

**Option B: Add session_closed flag in orderGroups from o.table_session status**

- Pro: No extra query if session status is denormalized on orders
- Con: Orders don't carry session status — would need joined data or separate query
- Con: Effectively same as Option A but more complex

**Option C: Check all orders .every(o => o.status === 'closed' || 'cancelled') → completed**

- Pro: No extra query
- Pro: Simple logic change
- Con: Doesn't handle case where closeSession partially failed (some orders closed, some not)
- Con: Doesn't handle the "served but waiter hasn't clicked close yet" intent (S267 explicitly wants served-but-not-closed in Active)

**RECOMMENDATION: Option A is the most robust.**

Cost: 1 additional `TableSession.filter({ status: 'open' })` query. This returns only open sessions (small set), runs at same polling interval as orders/requests. The override logic is straightforward:

```jsx
// Pseudo-code for filteredGroups override:
const sessionClosed = group.type === 'table' && !openSessionByTableId[group.id];
if (sessionClosed) {
  // Force to completed regardless of order/request status
  return activeTab === 'completed';
}
// ... existing logic
```

However, this requires knowing which session maps to which table. Since orderGroups groups by tableId, and TableSession has a `table` field, the mapping is: `openSessionByTableId = Object.fromEntries(openSessions.map(s => [getLinkId(s.table), s]))`.

**IMPORTANT caveat:** This only works if there's at most one open session per table (which should be enforced by openSession/closeSession logic). If multiple open sessions exist for a table, this needs additional handling.

### 1.6 Regression risks

1. **Tab counts desync:** If `tabCounts` doesn't use the same override logic as `filteredGroups`, the count badge will show wrong numbers. Both memos must be updated together.

2. **Sort order disruption:** `sortedGroups` (line 3792-3803) sorts by oldest unaccepted order. If a closed-session group is forcibly moved to completed, it will be sorted among completed groups. Need to verify `v2SortedGroups` (which uses `filteredGroups` output) handles this correctly.

3. **Re-open table UX:** If a new session is opened for the same table while the old session's orders are still in `activeOrders` (they have `status: 'closed'` which passes `activeOrders` filter at line 3583), the table would have a mix of old closed orders and new active orders. The session override would correctly show it as Active (new open session exists), but the old closed orders would be mixed in the same group (Fix C territory).

### 1.7 Test plan

1. **Close table → verify tab switch:** Open table with served orders → click "Close table" → confirm → verify table disappears from Active tab and appears in Completed tab within 2 seconds.

2. **Reload after close:** Close table → force reload page (F5) → verify closed table is in Completed tab, not Active.

3. **Swipe tabs:** After closing table, switch between Active/Completed tabs multiple times → verify counts are correct and table appears consistently in Completed.

4. **Reopen + new order:** After closing table, create new TableSession for same table → place new order → verify new group appears in Active tab with only new session's orders (also validates Fix C).

### 2.4 Diagnosis: filter format mismatch

**closeSession filters by:** `ServiceRequest.filter({ table: tableId })` where `tableId` is a string ID.

**ServiceRequest is created with:** `table: currentTableId` (string ID).

**On the surface, this looks correct** — both use string IDs. However, there's a critical B44 platform subtlety:

**B44 link fields:** When B44 stores a link field (like `table` which links to a Table entity), the stored value may be:
- A string ID (as created)
- A link object `{ id: "...", _type: "Table" }` (B44 internal format)

When using `.filter({ table: tableId })`, B44's backend should resolve the link and match. But there are known issues with B44 link field filtering:

1. **If `table` is stored as a link object but filter uses a string** — B44 may not match.
2. **If `table` field is not indexed** — filter returns empty.

**However, the actual test failure report says:** "after closing table, it stayed 'active'" (S286 testing). This could mean:

a. `ServiceRequest.filter({ table: tableId })` returned empty (filter doesn't work for this field)
b. The filter works but the update didn't propagate (timing)
c. `tableId` passed to `closeSession` was `undefined` or wrong format

**Checking the call chain:**
- SOM line 4148: `const { sessionId, tableId } = closeTableConfirm;`
- Line 4142: `setCloseTableConfirm({ sessionId, tableId, tableName: ... })`
- Line 4139: `handleCloseTable = (tableSessionField, tableName, tableId) => { ... }`
- Called via `onCloseTable` prop → OrderGroupCard line 2167: `onCloseTable(sessionId, identifier, group.id)`
  - `sessionId` = `getLinkId(order.table_session)` from line 2165
  - `identifier` = table display name
  - `group.id` = tableId from orderGroups (line 3754: `id: tableId` where `tableId = getLinkId(o.table)`)

So `tableId` in `closeSession` = `getLinkId(o.table)` = normalized string ID. This matches the creation format.

**Most likely root cause: B44 `.filter()` on link fields may require the field to be the link target ID in a specific format, or the `table` field on ServiceRequest may not be filterable server-side.**

**Alternative hypothesis:** The `table` field in ServiceRequest is a **link field** (not a plain string), and B44's `.filter()` on link fields requires passing the ID as-is (string), which should work. But if B44 internally stores it as `{ id: "abc123" }` and the filter tries exact match on `"abc123"`, it may fail.

**Diagnostic recommendation:** Add console.log to `closeSession` to log `requests` array length and `tableId` value. In production, the most reliable fix is to filter by `table_session` instead:

### 2.5 Recommended fix

**Primary fix: Filter by table_session instead of table**

```jsx
// sessionHelpers.js, line 176-177
// BEFORE:
if (tableId) {
  const requests = await base44.entities.ServiceRequest.filter({ table: tableId });

// AFTER:
if (sessionId) {
  // Filter by session — more reliable than table link field
  const requests = await base44.entities.ServiceRequest.filter({ table_session: sessionId });
```

**Rationale:**
- ServiceRequest is created with `table_session: tableSession?.id` (line 2995 PublicMenu)
- `sessionId` is the first parameter of `closeSession(sessionId, tableId)` — always available
- Filtering by `table_session` is semantically more correct: we want to close requests for THIS session, not all requests for the table (there might be requests from a previous session)
- `table_session` is a string ID, same as what's stored → filter should work

**Fallback fix: Try both filters**

```jsx
if (tableId || sessionId) {
  let requests = [];
  if (sessionId) {
    requests = await base44.entities.ServiceRequest.filter({ table_session: sessionId });
  }
  if (requests.length === 0 && tableId) {
    // Fallback: try by table (for requests created without session)
    requests = await base44.entities.ServiceRequest.filter({ table: tableId });
  }
  const openRequests = requests.filter(r => !['done', 'cancelled'].includes(r.status));
  // ... close loop
}
```

### 2.6 Regression risks

1. **Requests without table_session:** If older ServiceRequests were created before `table_session` field existed (or PublicMenu didn't pass it), filtering by `table_session` would miss them. The fallback fix handles this.

2. **Multiple sessions for same table:** If we filter by `table` instead of `table_session`, we'd close requests from ALL sessions of that table, not just the current one. This is actually desired behavior for closeSession (close everything), but could be surprising if a new session was opened while old requests lingered.

3. **Rate limiting:** The sequential update loop with 120ms delay could hit rate limits if there are many open requests. Current code handles up to ~50 requests within 6 seconds. If table has more (unlikely but possible), timeout could occur.

### 3.3 Recommended approach

**Option A: Composite key `${tableId}:${sessionId}`**

```jsx

    const tableId = getLinkId(o.table);
    const sessionId = getLinkId(o.table_session);
```

### 3.4 UX impact: re-opened table with unclosed orders from old session

**Current behavior (buggy):** All orders from both sessions appear in one group. Waiter sees old served/closed orders mixed with new active orders. The group stays Active because new orders are active.

**After Fix C:** Orders from closed session are filtered out of `visibleOrders`→`orderGroups`. Only the new session's orders appear. The old orders are gone from the waiter's view.

**Edge case:** What if the old session has orders that were NOT closed (closeSession partially failed)? These orders would have `table_session` pointing to a now-closed session → `openSessionIds.has(sessionId)` = false → filtered out. This is correct behavior: even if individual orders weren't properly closed, the session is closed, so they shouldn't appear in Active.

**Edge case:** What if a waiter wants to see what the previous guest at this table ordered? They lose this visibility. However, this is an intentional design choice — the Completed tab + served orders section already provides historical visibility.

## Section 5 — Risks Outside Scope

1. **activeOrders includes closed orders:** Line 3582-3583 — orders at finish stage with `status: 'closed'` pass the filter (`o.status !== 'cancelled'` is true). This means closed orders remain in `activeOrders` and `visibleOrders`, creating phantom groups even after session close until they age past shiftCutoff.

2. **Polling interval amplifies stale data:** Other SOM instances (other waiters) won't see closed table until their next poll cycle. If polling interval is 10s+, table shows Active for up to 10s on other devices.

3. **closeSession has no error handling for partial failures:** If order bulk-close succeeds but ServiceRequest close fails (or vice versa), the session is in an inconsistent state. No rollback mechanism exists.

4. **ServiceRequest.filter({ table: tableId }) returns ALL requests for the table across all sessions.** If table has been through multiple sessions, closeSession would mark ALL requests as done, not just current session's. This is usually OK but could mask data for analytics.

5. **expandedGroups Set uses group.id (= tableId).** If Fix C changes the grouping key to composite `tableId:sessionId`, the Set would need updating. All `expandedGroups.has(group.id)` checks and `toggleGroupExpand(groupId)` calls would break if group.id format changes.
