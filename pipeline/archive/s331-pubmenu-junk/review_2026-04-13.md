## Prompt Design Review: CartView Redesign Prompt

### Scope
This is a review of the implementation prompt design, not a code review of the final UI.

I could not retrieve the raw branch task body from tracked files or git metadata in this workspace. The findings below are based on:
- the current 1163-line `pages/PublicMenu/CartView.jsx`
- the upstream contract in `pages/PublicMenu/x.jsx`
- the 7-fix summary provided in the request
- earlier prompt-review artifacts already saved for `PublicMenu`

### Findings

- [P1] Fix 1 (`collapse 5 status buckets to 2 groups`) is under-specified unless the prompt names the raw status mapping and the grouping source. The current file still builds `statusBuckets` from raw `order.status` values in `CartView.jsx:402`, while the expanded-state shape still assumes five buckets in `CartView.jsx:93-99`. A prompt that talks only in UI labels like "served/ordered" or "2 groups" without explicitly mapping `served/completed` vs `ready/in_progress/accepted/new` is likely to produce the wrong filters. The prompt should include a mapping table and say "group by raw `order.status`, not by display labels".

- [P1] Fix 3 (`submit feedback state A2`) cannot be implemented in `CartView.jsx` alone unless the prompt explicitly widens scope to `x.jsx` and defines the new state contract. The actual rating mutation lives in `pages/PublicMenu/x.jsx:3297-3330`, where `handleRateDish` only toggles `ratingSavingByItemId[itemId]` between `true` and `false`, rolls back `draftRatings` on failure, and never exposes any `A2` or richer tri-state submit object. `CartView.jsx` only receives those props from the parent. If the prompt names a new `A2` state without defining where it comes from, the snippet is incomplete or uses variables that do not exist.

- [P1] Fix 5 (`remove split-type`) is stale against the current file shape and creates scope risk. The split-order UI is already gone from `CartView.jsx`, but `splitType` and `setSplitType` are still accepted in `CartView.jsx:32-33` and still passed from `pages/PublicMenu/x.jsx:4771-4772`. A prompt that simply says "remove split-type" does not tell the implementer whether to delete only dead props from `CartView.jsx`, keep the parent API untouched, or remove the parent prop plumbing in `x.jsx` too. That is a safety risk because it invites unintended file changes outside the target component.

- [P1] Fix 2 (`money only in header`) is ambiguous because money is currently rendered in several different semantic places, not one. The current component shows money in the header order sum (`CartView.jsx:417`, rendered at `713-720`), the table-total card (`CartView.jsx:842`), each status-group subtotal, and the new-order subtotal. If the prompt only says "money only in header", it does not tell the implementer whether the table-total card survives, whether the cart subtotal still belongs in the new-order section, or whether other-guests/table amounts are exempt. The prompt needs a keep/remove matrix for every money surface.

- [P2] Fix 4 (`auto-collapse logic`) needs transition semantics, not just a desired end state. The current effect collapses sections only on the boolean transition to `cart.length > 0` in `CartView.jsx:96-110`. The prompt should say whether collapse happens only on `0 -> >0`, on every cart mutation, and whether groups should reopen automatically when the cart goes back to empty. Without that, a naive implementation will keep fighting user interaction and repeatedly collapse sections after every quantity change.

- [P2] Fix 6 (`qty display`) is ambiguous because quantity appears in multiple render paths with different rules today. Grouped status rows, cart rows, and table-orders rows do not share one formatting rule. The prompt needs to say exactly which surfaces change; otherwise the implementer will almost certainly patch one path and miss the others.

- [P2] Fix 7 (`compact header`) is incomplete unless the prompt defines the mobile and interaction edge cases. The current header contains an optional bell action, inline guest-name edit mode, a second line for order count plus sum, and a close action guarded by `isSubmitting` in `CartView.jsx:673-734`. A prompt that only says "make header compact" but does not define behavior for long table names, long guest names, no-bell layout, edit mode, and in-flight submit close-lock leaves too much room for subjective edits and regression.

- [P2] Post-fix checks are not sufficient unless they verify both UI states and the parent contract. For this redesign, a good verification block should explicitly cover:
  - cart empty vs cart non-empty
  - served-only/history-only state
  - rating save, fail, and re-entry flows
  - header layout with long names on narrow mobile width
  - other-guests/table-total behavior
  - `x.jsx` still matching the final `CartView` prop list
  - no leftover dead state/props such as `showRewardEmailForm` (`CartView.jsx:114`), `splitType`, and `myBill`
If the prompt only asks for grep checks or a generic "page loads", that is too weak for a structural `CartView` rewrite.

### Summary
- Total findings: 8
- Severity split: P1: 4, P2: 4
- Most important prompt changes:
  - define exact raw status-to-group mapping for Fix 1
  - define whether Fix 3 is `CartView.jsx` only or also requires `x.jsx`
  - scope-lock Fix 5 so split-type cleanup does not spill into unintended files
  - replace "money only in header" with a precise surface-by-surface rule

### Practical Recommendation
Before implementation, the prompt should be revised to add:
- a status mapping table
- a file-scope section (`CartView.jsx` only vs `CartView.jsx` + `x.jsx`)
- an explicit keep/remove matrix for header, table-total, status-group subtotal, new-order subtotal, and footer money
- transition rules for auto-collapse
- a verification checklist that includes parent-contract checks, not only UI screenshots or greps
