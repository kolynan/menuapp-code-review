### Q1: What 2 improvements should be made to `cleanup_stale_git_locks`?

**CC Position:**
1. **Dry-run mode + log lock age/PID before deletion** — Add `dry_run=False` parameter so the function can be called speculatively (e.g., health-check loops) without side effects. Log each lock's age and (if recoverable) owning PID before deleting, improving post-mortem debugging.

**Improvement 1 (Dry-run + age logging):**
- The function already logs age: `f'(age {age_sec:.0f}s, repo={repo_dir.name})'` on line ~525. So "log lock age" is already implemented.
- PID recovery from lock file content is git-specific (git writes PID into `index.lock`). This is a valid small win — `lock_path.read_text().strip()` before unlink, log the PID, and optionally check if that PID is alive (`os.kill(pid, 0)`).

- **Verdict:** Partially valid. The PID-check portion is the real value-add; age logging is already done; dry-run is speculative.

**Improvement 2 (TOCTOU race guard):**
- The TOCTOU window is real but extremely narrow on Windows/NTFS. In practice, two git processes rarely target the same lock path within milliseconds in this pipeline (worktrees isolate most git ops).
- The "check if file reappeared after delete" approach adds complexity for a scenario that hasn't occurred in production (the original bug KB-159 was a crash leaving an orphan, not a race).
- The `git status` subprocess check could itself deadlock if the lock is stuck.
- **Verdict:** Theoretically sound, but over-engineered for the current failure mode. A simpler guard: read lock file PID → check if PID is alive → skip if alive. This covers the real risk without TOCTOU timing games.

**Resolution:** Accept CC's proposals with refinements:

### Final Recommended Improvements

**Improvement A: PID-aware deletion (refined from CC #1)**
Before unlinking, read the lock file content to extract the PID (git writes it there). If PID is alive (`os.kill(pid, 0)` or Windows equivalent), skip deletion and log a warning. This prevents deleting locks held by slow-but-alive git processes — the actual dangerous scenario.

**Improvement B: Telegram alert on lock removal (new — derived from pipeline context)**
The function returns `removed: list[str]` but the callers don't always surface this to Telegram. Given that stale locks indicate crashed processes (KB-159 scenario), every lock removal should trigger a TG alert (not just a log warning). This ensures operators know a crash-cleanup happened, which may indicate a deeper stability issue worth investigating.

*Note: CC's dry-run proposal is valid but lower priority — park for future if health-check loop is added. CC's TOCTOU race guard is theoretically correct but over-engineered for current usage; PID-alive check (Improvement A) covers the real risk more simply.*
