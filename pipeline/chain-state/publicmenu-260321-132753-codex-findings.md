# Codex Writer Findings — PublicMenu
Chain: publicmenu-260321-132753

## Findings
1. [P1] Inline table verification still renders inside the cart drawer — `CartView.jsx` still shows the amber verification block and auto-verifies as soon as the code is fully entered, so guests are asked for table code before tapping "Отправить официанту". FIX: remove the inline verification UI and auto-verify block from `CartView.jsx`; keep table confirmation exclusively in the post-submit bottom sheet.
2. [P1] Table-confirmation sheet hardcodes 4-digit codes — `x.jsx` renders 4 slots and slices input to 4 (`Array.from({ length: 4 })`, `.slice(0, 4)`, `maxLength={4}`), while `CartView.jsx` already supports partner-configurable `table_code_length`. Partners with 3/5-8 digit codes cannot complete the flow. FIX: drive the sheet from the same `tableCodeLength` value used in `CartView.jsx` and remove hardcoded `4`/`0000` values.
3. [P2] Bottom sheet still skips the explicit confirm action — `x.jsx` auto-submits after verification through the `pendingSubmitRef` effect, but the sheet itself has no primary "Подтвердить и отправить" CTA. The flow is still implicit and easy to miss on mobile. FIX: add an explicit primary button in the sheet and submit from that button instead of hidden auto-submit.
4. [P1] Partner lookup hides backend failures as "restaurant not found" — `x.jsx` swallows both primary and fallback query errors and returns `null`, so network/server failures fall into the `!partner` UI instead of a retryable error state. FIX: preserve unexpected query errors, expose them from `useQuery`, and render a distinct backend-failure message with retry.
5. [P1] Hall session restore never retries after a transient error — `useTableSession.jsx` sets `didAttemptRestoreRef.current = true` before async restore work and never resets it on failure. One temporary read error leaves `tableSession` unset, polling never starts, and the hall visit stays empty until refresh. FIX: reset the guard on failure or replace it with bounded retry/backoff plus a visible restore error state.
6. [P2] Bill and table totals use raw values in arithmetic — `useTableSession.jsx` accumulates `item.line_total` and `o.total_amount` without numeric coercion in split-bill and `tableTotal` calculations. If Base44 returns numeric strings, totals concatenate instead of sum. FIX: wrap every accumulated monetary field in `Number(...) || 0` before adding.
7. [P1] Pickup/delivery checkout still drops loyalty UI — `x.jsx` passes loyalty props into `CheckoutView`, but `CheckoutView.jsx` neither destructures them nor renders any loyalty/email/redeem UI. Discounts and points remain unavailable outside the hall drawer flow. FIX: wire the passed loyalty props into `CheckoutView.jsx` and render the loyalty block for pickup/delivery checkout.
8. [P1] Hall visit-state UI is still reduced to generic modes — `x.jsx` computes only `cart | myBill | tableOrders | cartEmpty`, and `CartView.jsx` keeps a single `cart.send_to_waiter` submit label. Post-send/closed/paid/draft states from the visit lifecycle are still missing. FIX: introduce a shared visit-state enum and drive both the sticky bar and drawer CTA/copy from that full matrix.
9. [P3] Mobile stepper controls still have accessibility/i18n gaps — `MenuView.jsx` list-mode quantity buttons have no labels, and `CheckoutView.jsx` uses hardcoded English `aria-label` values plus 32px controls that miss the 44px mobile touch target. FIX: localize all stepper labels with `t(...)` and raise the controls to minimum 44x44 hit areas.
