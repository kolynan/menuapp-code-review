# Codex Writer Findings - PublicMenu Chain: publicmenu-260323-124313-bcbe
## Findings
1. [P1] `popstate` closes the lower cart drawer before the top table-code sheet - In `pages/PublicMenu/x.jsx:2354-2364`, the back handler checks `drawerMode === 'cart'` before `showTableConfirmSheet`. When both layers are open, the first Android Back consumes the top sheet's history entry but closes the lower order-list drawer instead, which is the exact wrong-order regression described in Fix 1. FIX: make the back handler state-aware and close the highest-priority active layer first, ideally by matching `event.state.sheet` to the exact close action so `tableConfirm` is popped before `cart`.
2. [P1] Programmatic sheet closes leave stale history entries behind - The cart and table-code sheet close paths in `pages/PublicMenu/x.jsx:2124-2129`, `pages/PublicMenu/x.jsx:3375-3377`, `pages/PublicMenu/x.jsx:3444`, and `pages/PublicMenu/x.jsx:3462-3469` only call `setDrawerMode(null)` / `setShowTableConfirmSheet(false)`. Because none of these paths call `history.back()`, dismissing a sheet via chevron, overlay tap, or successful verification leaves its pushed history entry on the stack, so later Back presses pop stale entries and require extra presses or close the wrong remaining layer. FIX: centralize non-`popstate` sheet closes and pair each one with a guarded `history.back()` for the matching pushed state instead of mutating React state alone.
3. [P1] The history stack is incomplete for the required overlay priority list - Only cart and table-confirm ever push history entries (`pages/PublicMenu/x.jsx:2682`, `pages/PublicMenu/x.jsx:3355`, `pages/PublicMenu/x.jsx:3640`). Checkout, confirmation, and order-status transitions only change `view` / URL replacement (`pages/PublicMenu/x.jsx:1464-1485`, `pages/PublicMenu/x.jsx:3040-3041`), so Android Back cannot uniformly close `checkout`, `confirmation`, or `orderstatus` as topmost layers. That means the required `table code > checkout/confirmation > order status > cart > menu` stack contract is still not implemented. FIX: assign every overlay/view in that priority list a unique pushed history state on open and close the popped state in `popstate` instead of inferring behavior from a couple of booleans.
