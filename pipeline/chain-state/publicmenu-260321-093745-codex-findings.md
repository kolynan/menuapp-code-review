# Codex Writer Findings — PublicMenu
Chain: publicmenu-260321-093745

## Findings
1. [P0] Loyalty side effects run before order items exist - In both submit paths, `Order.create()` is followed by loyalty redeem/update before `OrderItem.bulkCreate()` (`pages/PublicMenu/base/x.jsx:2436-2470`, `pages/PublicMenu/base/x.jsx:2812-2843`). If item creation fails, points are already spent and the order can be left partial or empty. FIX: create all order items first, then apply loyalty side effects; if Base44 cannot do a transaction, treat item creation as the commit point and only debit loyalty after it succeeds.
2. [P1] Hall confirmation shows raw cart total, not real order total - Hall orders are persisted with `total_amount: finalTotal` (`pages/PublicMenu/base/x.jsx:2420`), but the confirmation screen is fed `cart.reduce(...)` (`pages/PublicMenu/base/x.jsx:2540-2559`). Any loyalty discount or redeemed points makes the guest-facing total wrong. FIX: pass `finalTotal` or the created order's stored `total_amount` into the hall confirmation payload.
3. [P1] Mode switches keep invalid cart items alive - `handleModeChange()` only flips `orderMode` and URL/localStorage (`pages/PublicMenu/base/x.jsx:2205-2225`); the cart is never revalidated, and cart entries only store `dishId`, `name`, `price`, `quantity` (`pages/PublicMenu/base/x.jsx:2229-2243`). A hall-only dish can stay in cart and be submitted from pickup/delivery after it disappears from the menu. FIX: on every mode change, revalidate cart items against mode availability and either drop invalid items with a toast or block checkout until the cart is cleaned.
4. [P1] Pickup/delivery checkout drops loyalty and discount controls - `x.jsx` passes loyalty props into `CheckoutView` (`pages/PublicMenu/base/x.jsx:3174-3197`), but `CheckoutView` does not accept or render them (`pages/PublicMenu/base/CheckoutView.jsx:7-31`, `pages/PublicMenu/base/CheckoutView.jsx:88-150`). Guests cannot enter email, see balance, redeem points, or understand discount application in non-hall checkout. FIX: plumb the passed loyalty props through `CheckoutView` and render the same loyalty/discount block used for hall checkout.
5. [P1] Visit-state UI is still count-driven instead of state-driven - `CartView` always renders the same section stack when data exists (`pages/PublicMenu/base/CartView.jsx:557-780`, `pages/PublicMenu/base/CartView.jsx:783-1277`), and the sticky footer exists only when `cart.length > 0` (`pages/PublicMenu/base/CartView.jsx:1245-1277`). In `x.jsx`, `hallStickyMode` has only `cart`, `myBill`, `tableOrders`, `cartEmpty` and is derived from counts (`pages/PublicMenu/base/x.jsx:2056-2115`). This misses the required visit/draft matrix: no bottom CTA for sent-orders-without-draft, no unpaid secondary line for new-draft-after-send, no paid/closed fade-out behavior. `useTableSession` also merges draft into table total and never filters paid orders (`pages/PublicMenu/base/useTableSession.jsx:722-747`, `pages/PublicMenu/base/useTableSession.jsx:784-788`). FIX: derive explicit visit state from session/order/payment status, compute unpaid sent totals separately from draft totals, and render drawer/sticky CTAs from that state matrix instead of `cart.length` or order counts.
6. [P2] Terracotta palette is still incomplete - Primary UI elements still use indigo or green in the reviewed files: `MenuView.jsx` prices, add buttons, layout toggle, and loader (`pages/PublicMenu/base/MenuView.jsx:85`, `pages/PublicMenu/base/MenuView.jsx:104`, `pages/PublicMenu/base/MenuView.jsx:162`, `pages/PublicMenu/base/MenuView.jsx:203`, `pages/PublicMenu/base/MenuView.jsx:229`, `pages/PublicMenu/base/MenuView.jsx:240`, `pages/PublicMenu/base/MenuView.jsx:252`), `ModeTabs.jsx` active tab (`pages/PublicMenu/base/ModeTabs.jsx:33`), and `CheckoutView.jsx` link, total, and submit button (`pages/PublicMenu/base/CheckoutView.jsx:37`, `pages/PublicMenu/base/CheckoutView.jsx:76`, `pages/PublicMenu/base/CheckoutView.jsx:173`). FIX: replace remaining `indigo` and `green` primary styles with the terracotta token/color system used elsewhere.
7. [P2] Mobile-first controls are still too small in key flows - Several tap targets are below 44px or icon-only in the reviewed mobile UI: list add button `40x40` and unlabeled list steppers in `MenuView.jsx` (`pages/PublicMenu/base/MenuView.jsx:102-121`), tile steppers `32x32` (`pages/PublicMenu/base/MenuView.jsx:168-183`), tiny checkout steppers in `CheckoutView.jsx` (`pages/PublicMenu/base/CheckoutView.jsx:57-64`), and thin mode tabs in `ModeTabs.jsx` (`pages/PublicMenu/base/ModeTabs.jsx:27-35`). These are hard to hit on phones and regress the mobile-first brief. FIX: raise all primary tap targets to at least `44x44`, add `aria-label`s to list-mode quantity buttons, and avoid icon-only controls for primary actions where space allows.
8. [P2] Localization fallback is hardcoded in Russian inside page code - The page-level fallback path is a hardcoded Russian map in `x.jsx` (`pages/PublicMenu/base/x.jsx:325-506`), and `CartView.jsx` repeats many `tr('key', 'Русский текст')` fallbacks (`pages/PublicMenu/base/CartView.jsx:241-266`, `pages/PublicMenu/base/CartView.jsx:407-1273`). If any key is missing in EN/KZ, the UI silently mixes the selected locale with Russian copy instead of a locale-aware fallback. FIX: move fallback copy into translation catalogs, make fallback language configurable, and remove hardcoded per-component Russian strings.
