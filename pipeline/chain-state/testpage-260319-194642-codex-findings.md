# Codex Writer Findings — TestPage

## Findings
1. [P0] Broken i18n hook import path — [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:2) imports `useI18n` from `@/components/useI18n`, but the repo consistently imports it from `@/components/i18n`, and no `useI18n` module exists in this checkout. If the alias resolves strictly, the page fails before first render. FIX: change the import to `@/components/i18n` so the page uses the same module path as the rest of the app.
2. [P1] Raw exception messages leak untranslated browser text — [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:25) and [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:39) store `err.message`, and [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:51) renders it directly. Network/CORS/JSON failures can therefore show browser strings like `Failed to fetch`, which breaks Base44 i18n rules and exposes low-level errors to guests. FIX: map unexpected failures to translated UI messages such as `t("test_page.fetch_failed")`, `t("test_page.delete_failed")`, or `t("common.error")` instead of rendering raw exception text.
3. [P1] `test_page.*` translation keys are missing from the repo catalog — [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:22), [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:37), [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:50), and [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:52) use `test_page.fetch_failed`, `test_page.delete_failed`, `test_page.title`, and `test_page.no_items`, but those keys do not appear in the repo’s translation sources in this snapshot. Depending on how `useI18n` handles misses, users will see raw keys or empty copy. FIX: add the `test_page.*` keys to the translation catalog and provide explicit fallback text if this page must remain readable before translations are synced.
4. [P1] Single `deletingId` state breaks overlapping deletes — [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:12), [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:34), [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:42), and [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:58) assume only one delete request can be in flight. On mobile, tapping item A and then item B swaps the spinner to B, re-enables A, and the first request can clear the loading state while the second is still pending. FIX: track deleting state per item id, or disable all delete actions while any delete request is active.
5. [P2] `useEffect` references a later `const` function — [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:14) calls `fetchItems()` before the `const fetchItems = async () => {}` declaration at [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:18). It works today only because the effect runs after render, but it violates the repo’s TDZ-safety rule and is a brittle React pattern. FIX: move `fetchItems` above the effect or inline the async loader inside the effect.
6. [P2] List row layout is not mobile-safe for long item names — [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:54) renders plain text next to the delete button inside `justify-between`, but the name has no `flex-1`, `min-w-0`, wrapping, or truncation. Longer dish names can push the delete action off-screen or collide with it on narrow phones. FIX: wrap the name in its own flex child and add `flex-1 min-w-0` plus `break-words` or truncation classes.
7. [P2] Error state has no recovery action — [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:51) shows the error, but there is no retry button and the loading branch at [`pages/TestPage/base/testpage.jsx`](C:/Dev/menuapp-code-review/pages/TestPage/base/testpage.jsx:46) offers no recovery path either. In a QR-menu flow on unstable restaurant Wi‑Fi, a transient fetch failure leaves the user stuck until they manually reload the page. FIX: render a retry action that calls `fetchItems()` and keep it available in the error state.
