# Codex Writer Findings — PublicMenu
Chain: publicmenu-260321-173133

## Findings
1. [P1] Hall submit validates table before opening confirmation sheet - `x.jsx:2334-2350` requires `currentTableId` in hall mode, but `x.jsx:2628-2632` opens the confirmation Bottom Sheet only after `validate()` passes. On `/x?...mode=hall` with no verified table, the click dies at `newErrors.tableSelection`; there is no rendered `tableSelection` error in the hall drawer, so the user sees nothing. FIX: move the `!isTableVerified` gate above `validate()`, or skip hall `tableSelection` validation until after the Bottom Sheet verification flow.
2. [P1] Partner lookup converts backend failures into "restaurant not found" - `x.jsx:1322-1346` catches both lookup attempts and returns `null`, and `x.jsx:3035-3040` always renders `error.partner_not_found` when `partner` is null. Transient API failures are misreported as a bad QR link and the query never retries because it resolves successfully. FIX: let request errors throw (or store them separately), use React Query's error state, and render a retryable backend error distinct from true not-found.
3. [P1] Pickup/delivery checkout drops loyalty and discount flow entirely - `x.jsx:3220-3263` passes loyalty props into `CheckoutView`, but `CheckoutView.jsx:7-32` does not accept them and `CheckoutView.jsx:88-149` renders only contact/comment fields. Guests cannot enter loyalty email, see balance or discounts, or redeem points in pickup/delivery even though submit logic still supports those fields in `x.jsx`. FIX: extend `CheckoutView` to consume the passed loyalty and discount props and render the same email, balance, and redeem UI for non-hall checkout.
4. [P1] Session restore stops forever after a single transient failure - `useTableSession.jsx:265-487` sets `didAttemptRestoreRef.current = true` before the async restore work, then swallows all errors in the outer `catch` without resetting it. If any restore request fails once, the effect never retries until the table changes or the page is reloaded. FIX: reset the attempt flag on failure, or only mark restore as attempted after successful completion and add retry/backoff handling for recoverable errors.
5. [P1] Hall StickyCartBar ignores visit lifecycle and bill state - `x.jsx:2101-2144` derives `hallStickyMode` only from `cart`, `myOrders`, and `sessionOrders`, while bill state exists separately in `x.jsx:1314-1315`, `x.jsx:2306-2310`, and `x.jsx:2984-3002` and is never consulted here. That prevents lifecycle-specific CTA, copy, and visibility for requested bill, closed visit, paid visit, and other post-order hall states. FIX: compute sticky-bar mode from `tableSession`, `billRequested`, `billCooldown`, and visit or payment status before choosing label, amount, and visibility.
6. [P2] Table-code verification is split between hidden CartView state and visible Bottom Sheet - `CartView.jsx:89-177` still owns code attempts and lockout state and `CartView.jsx:180-206` auto-calls `verifyTableCode`, but the live Bottom Sheet in `x.jsx:3383-3478` does not read that state and its CTA directly calls `verifyTableCode` again. The visible sheet therefore has no cooldown or countdown UI and verification behavior comes from two components at once. FIX: move code attempts, lockout, and auto-submit state into `x.jsx` or the shared verification hook, remove the hidden CartView verification effects, and let the sheet render and enforce the single source of truth.
7. [P2] Mobile tile layout ignores partner's configured mobile grid size - `MenuView.jsx:42-49` computes `mobileCols` from `partner.menu_grid_mobile`, but `MenuView.jsx:284` hardcodes `grid-cols-2` whenever `isMobile` is true. Partners configured for a 1-column mobile grid still get 2 columns. FIX: use `MOBILE_GRID[mobileCols]` or `gridColsClass` in the mobile tile branch instead of hardcoding `grid-cols-2`.
8. [P3] Several user-facing controls still miss required i18n or accessibility wiring - examples: `CheckoutView.jsx:58-63` hardcodes English `aria-label`s (`"Decrease"`, `"Increase"`), `MenuView.jsx:102-125` list-mode quantity buttons have no `aria-label`s at all, and `CartView.jsx:467-485` has unlabeled edit and close buttons plus a hardcoded reward-email placeholder at `CartView.jsx:524`. This creates mixed-language UI and weaker screen-reader behavior. FIX: localize all placeholders and labels with `t()` or `tr()` and add explicit `aria-label`s to every icon-only control.
