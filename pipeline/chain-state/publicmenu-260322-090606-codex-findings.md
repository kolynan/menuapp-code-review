## Findings

[P1] Partner lookup collapses backend failures into “restaurant not found” — In `x.jsx:1322-1345`, both partner lookups swallow exceptions and return `null`; then `x.jsx:3037-3042` always renders `error.partner_not_found`. A transient Base44/network failure is misreported as a bad QR link, so guests get no retryable error state. FIX: keep lookup exceptions separate from a true empty result and render a retry-capable backend error UI instead of always showing `error.partner_not_found`.
2. That hides server/network failures and tells guests their link is invalid when the backend may just be unavailable. FIX: split `orderError` from `!order`, show a retryable fetch-error state for query failures, and reserve `order_status.not_found` for confirmed empty results.
3. [P1] Hall StickyCartBar ignores visit lifecycle state — `x.jsx:2101-2144` derives the hall sticky bar from only four modes (`cart`, `myBill`, `tableOrders`, `cartEmpty`) based on cart/order counts. It never inspects visit lifecycle state such as closed/paid/draft-specific states, so CTA text, totals, and visibility are wrong once a visit moves past the basic “cart vs orders” cases. FIX: derive sticky-bar state from the full table session / bill lifecycle matrix and map label, amount, and visibility from that state instead of only from item counts.
4. [P2] Dynamic primary color is not implemented in `x.jsx` — The file never reads `partner.primary_color`, and hardcoded terracotta is still present at `x.jsx:751`, `x.jsx:770`, `x.jsx:1024`, `x.jsx:1156-1157`, `x.jsx:3190`, `x.jsx:3426`, and `x.jsx:3460`. That means the page still ignores partner-selected branding and also keeps the old terracotta fallback instead of the required `#1A1A1A`. FIX: define `primaryColor`/derived variants from `partner?.primary_color || '#1A1A1A'` near the main component state and replace every remaining `#B5543A`-based accent in this file with dynamic inline styles/props.
5. [P2] Auto-submit after table verification uses an untracked timeout — In `x.jsx:2091-2098`, the effect schedules `setTimeout(() => handleSubmitOrder(), 100)` with no cleanup and without tracking the timer handle. That is a React anti-pattern and can fire a stale submit if the component rerenders/unmounts or the user changes state during the 100 ms gap. FIX: store the timeout id in a ref, clear it in the effect cleanup, and invoke a stable submit callback only while the confirmation flow is still current.
6. [P3] Table-code confirmation sheet has no real focus feedback on mobile — The visible digit cells at `x.jsx:3424-3428` use `focus-within:border-[#B5543A]`, but the actual `<Input>` is rendered separately at `x.jsx:3433-3445`, outside those cells. On mobile, focusing the input never highlights the code boxes, so the active typing target is unclear in the bottom sheet. FIX: drive the cell highlight from explicit input-focus state, or move the focusable input inside the visual wrapper that is supposed to react to focus.
