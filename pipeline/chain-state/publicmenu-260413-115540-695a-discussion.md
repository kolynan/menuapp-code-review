## Analysis

### Q1: UI-паттерн для сообщения «Сервис не подключён»

**CC Position:** (b) Bottom sheet — компактный (30-40% экрана), 2-3 строки + dismiss. Toast слишком мимолётный для объяснительного текста. Inline ломает 3×2 grid и проблематичен при множественных тапах. Bottom sheet — стандартный мобильный паттерн (Uber Eats, Wolt, Glovo).

**Codex Position:** (c) Inline message — показывать объяснение прямо в drawer под нажатой кнопкой. Сохраняет контекст, не стакает вторую поверхность поверх drawer, не требует лишнего dismiss-действия.

**Status:** disagreement

**Resolution:** **Synthesizer: Bottom sheet (CC)** — с учётом аргументов обеих сторон.

Codex поднимает валидный аргумент: стакать bottom sheet поверх существующего drawer — не идеально. Однако при детальном анализе bottom sheet побеждает:

1. **Layout integrity.** UX v6.0 задаёт жёсткий grid 3×2 с системой состояний кнопок (idle → sending → active → urgent). Inline-сообщение под кнопкой ломает этот grid — кнопки смещаются, на 320px может появиться скролл. Если гость тапнет 2 кнопки, два inline-сообщения ещё больше разрушат layout.

2. **Заметность для upsell-цели.** Главная цель #180 — гость должен ЗАПОМНИТЬ что функция не работает и сказать ресторану. Bottom sheet заметнее inline-текста, который легко проскочить мимо.

3. **Стакинг drawer+sheet.** Help Drawer сам является bottom sheet, но это не проблема: bottom sheet поверх drawer — стандартный паттерн в мобильных приложениях (confirmation sheets в Apple Maps, Uber). Маленький sheet (30-40%) не перекрывает drawer полностью, а лёгкий dismiss (свайп/tap backdrop) возвращает в drawer.

4. **Единый компонент.** Bottom sheet — один компонент для всех 6+1 кнопок, не нужно управлять inline-состоянием для каждой кнопки отдельно.

**Nuance from Codex to keep:** если bottom sheet окажется слишком тяжёлым в реализации на Base44, inline — допустимый fallback с ограничением: показывать только 1 inline-сообщение, заменяя предыдущее при тапе на другую кнопку.

### Q2: Scope gate — какие кнопки блокировать?

**CC Position:** (a) Все 6 кнопок + «Другой запрос?» — полная блокировка Help Drawer. Все 6 — сервисные запросы к персоналу, все требуют бэкенд ServiceRequest. Частичная блокировка вводит в заблуждение.

**Codex Position:** (b) Только 4 «активные» (вызов, счёт, заказ, отзыв) — оставить навигационные/информационные бесплатными. Полная блокировка делает drawer «мёртвым», Free-план становится слишком слабым.

**Status:** disagreement

**Resolution:** **Synthesizer: Все 6 кнопок (CC)** — техническая реальность решает.

Ключевой факт из UX v6.0: все 6 кнопок Help Drawer — это именно сервисные запросы (Позвать, Счёт, Вода, Салфетки, Приборы, Убрать). Среди них НЕТ навигационных или информационных — все 6 отправляют ServiceRequest через бэкенд. На Free-плане бэкенд ServiceRequest не работает → ни одна кнопка технически не может выполнить свою задачу.

Codex предполагал наличие «навигационных/информационных» кнопок в drawer, но таких нет (reference files were missing in Codex's checkout). Разделение на «4 платных + 2 бесплатных» потребовало бы либо менять логику «Приборы»/«Убрать» (чтобы они работали без бэкенда, что невозможно), либо обманывать гостя (кнопка выглядит рабочей, но ничего не отправляет).

**Free-план всё равно имеет ценность:** QR-меню работает полностью. Help Drawer — это upsell-витрина, показывающая что ресторан МОЖЕТ подключить.

### Q3: Визуальный сигнал до тапа

**CC Position:** (a) Opacity 0.5 + 🔒 замок 12-14px в правом верхнем углу. Кнопки остаются тапабельными. Честность перед гостем — LMP (Spotify, Canva, Notion).

**Codex Position:** (a) Subtle lock + reduced emphasis, но предостережение: не димить слишком сильно, чтобы кнопка не выглядела disabled. Нужен accessibility tuning для контраста.

**Status:** agreed (с нюансом)

**Resolution:** **Agreed: Opacity 0.5-0.6 + 🔒 замок, кнопки тапабельные.** Принять нюанс Codex: opacity не ниже 0.5, чтобы сохранить affordance «это можно нажать». На 320px замок увеличить до 14-16px для читаемости.

Оба AI единогласны: вариант (b) «без отличия» — тёмный паттерн. Вариант (c) «отдельная секция» — избыточен при блокировке всех 6 кнопок.

**Implementation detail:** Tailwind `opacity-50` + абсолютно позиционированный `Lock` icon из lucide-react, `w-3.5 h-3.5 text-gray-400`. Кнопка сохраняет `cursor-pointer` и обычный tap target 44×44px.

### Q4: Текст сообщения

**CC Position:** (b) «Ресторан пока не подключил эту функцию. Вы можете обратиться к персоналу напрямую.» Заголовок: «Услуга не подключена». i18n ключи: `help_drawer.free_gate.title`, `help_drawer.free_gate.message`.

**Codex Position:** (b) «Эта функция пока не подключена рестораном. Вы можете обратиться к персоналу напрямую.» Нейтральное объяснение + альтернатива.

**Status:** agreed

**Resolution:** **Agreed: вариант (b) с альтернативой.** Формулировки CC и Codex почти идентичны, различие только в порядке слов. Финальная версия:

- **Заголовок:** «Услуга не подключена» / «Service not available»
- **Текст:** «Ресторан пока не подключил эту функцию. Вы можете обратиться к персоналу напрямую.» / «This restaurant hasn't activated this feature yet. You can ask the staff directly.»
- **i18n:** `help_drawer.free_gate.title`, `help_drawer.free_gate.message`

Оба AI согласны: вариант (a) слишком сухой (нет альтернативы), вариант (c) — вводит в заблуждение словом «временно».
