## Issues Found
1. [CRITICAL] Cancel-confirm design is not safe for `other` requests — The prompt introduces `cancelConfirmType` as a single type string and `handleSosCancel(type)` looks up `activeRequests.find(r => r.type === type)`. That is not enough for `other`, because the actual source supports multiple active `other` rows at `pages/PublicMenu/260408-01 PublicMenu x RELEASE.jsx:2007-2014`, and the proposed JSX also renders multiple `other` rows. If confirm is ever routed through this handler, it can pick the wrong row. The prompt also bypasses the confirm flow for `other` entirely by wiring the `other` row close button straight to `handleResolve('other', row.id)`, which conflicts with its own rule `non-red = instant, red = confirm panel`. PROMPT FIX: replace `cancelConfirmType` with an object such as `{ type, rowId }`, make `handleSosCancel(type, rowId)` resolve the exact row, wire `other` row cancel buttons through the same handler, and guard the confirm button so it no-ops if the target row no longer exists.
2. [CRITICAL] The prompt validates one file but tells the agent to edit a different one — All “verified” anchors are taken from `pages/PublicMenu/260408-01 PublicMenu x RELEASE.jsx`, but the prompt tells the implementation agent to edit only `pages/PublicMenu/x.jsx`. In this workspace those files are not byte-identical: the release snapshot is 5458 lines, while `pages/PublicMenu/x.jsx` is 5460 lines and contains trailing NUL padding that is not present in the release file. That creates a real safety risk for line-based editing, exact-copy instructions, and post-fix diff review. PROMPT FIX: either target the release file directly, or add an explicit first step to normalize `pages/PublicMenu/x.jsx` to the validated release snapshot and confirm the target file has no binary padding before any edits.
3. [MEDIUM] Some “verified” anchors are materially wrong — The drawer boundary anchors are mostly correct, but several important references are not. `handleResolve` starts at line 2455 and its `useCallback` closes at line 2493, not “~2470”. `HELP_URGENCY_GROUP` ends at line 1873, not “~1879”. `nonOtherTypes` is defined at line 2215, not 2216. The prompt also says the release file has 5459 lines, while the actual release snapshot has 5458. These are small individually, but the prompt relies heavily on exact placement. PROMPT FIX: correct the specific anchors above and prefer symbol-based anchors (“insert after the `handleResolve` callback closing `}, []);`”) over approximate line numbers wherever possible.
4. [MEDIUM] Legacy `menu` requests are still synchronized but become invisible in the new UI — The actual source still treats `menu` as a valid legacy help type in `HELP_REQUEST_TYPES` at `pages/PublicMenu/260408-01 PublicMenu x RELEASE.jsx:1835-1838` and in `nonOtherTypes` at `pages/PublicMenu/260408-01 PublicMenu x RELEASE.jsx:2215-2217`. That leaves a live edge case where an existing legacy `menu` request can remain in state but have no visible tile or cancel path in the new drawer. PROMPT FIX: explicitly decide the migration behavior for legacy `menu` requests: either filter and retire them before rendering, or render a temporary legacy row with cancel/retry behavior until old requests age out.
5. [MEDIUM] The proposed JSX violates the repo’s “no dynamic Tailwind classes” rule — The prompt tells the implementation agent to build `activeBg`, `activeBorder`, `labelColor`, `timerColor`, `xBg`, and `xColor` strings and interpolate them into `className`. That conflicts with the project rule to avoid dynamic Tailwind classes, and it invites style-review churn or missing styles if class extraction is strict. PROMPT FIX: instruct the agent to use an explicit urgency-to-class map with complete static class strings, e.g. `const SOS_URGENCY_STYLES = { red: '...', amber: '...', neutral: '...' }`, and reference those named variants directly.
6. [LOW] The cleanup instructions leave dead hooks behind for a reason that is technically inaccurate — The prompt says to keep `highlightedTicket` and `isTicketExpanded` “for hook order safety” while also removing their real consumers: old drawer JSX, `focusHelpRow`, and the post-send highlighting block. After those removals, both hooks become dead state. Keeping them does not protect React hook order across code versions, but it can create unused-variable noise and muddle the cleanup scope. PROMPT FIX: either remove those hooks entirely, or explicitly state that they are temporary compatibility shims and rename/comment them in a way that will not trip lint or confuse the next reviewer.
7. [MEDIUM] Post-fix validation is too weak for a 300-line JSX replacement — The prompt’s checks are mostly `grep` and line-count heuristics plus manual UI assertions. Those checks will not catch malformed JSX, a misplaced insertion after `handleResolve`, stale references like `setHighlightedTicket`, or a partially updated `x.jsx` that still contains binary garbage. PROMPT FIX: add at least one deterministic syntax-level validation step for the edited file, plus targeted symbol checks for `SOS_BUTTONS`, `handleSosCancel`, `cancelConfirmType`, and the absence of the removed helpers/refs after the rewrite.
