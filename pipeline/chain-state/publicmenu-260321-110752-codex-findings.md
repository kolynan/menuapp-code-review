## Findings
1. [P1] Hall submit still hard-blocks unknown tables ‚Äî `handleSubmitOrder()` returns `error.table_required` as soon as Hall mode has no `currentTableId`, and `renderHallCheckoutContent()` still renders the old pre-submit verification block. This preserves the ‚Äúenter table code before sending‚Äù behavior, so guests without the code are still stuck instead of getting the just-in-time confirmation bottom sheet described in the task context. FIX: replace the early return with a table-confirmation bottom sheet/intercept, continue submission after successful `verifyTableCode`, and remove the upfront mandatory verification path.
2. [P1] Partner lookup hides real backend failures behind ‚Äúrestaurant not found‚Äù ‚Äî the partner query catches both lookup attempts and returns `null` on any Base44/network error, then the page renders `error.partner_not_found`. A transient outage is misreported as a bad link and gives the guest no retry path. FIX: let transport/query failures throw, render a retryable error state for query errors, and reserve `partner_not_found` for a genuine empty result.
3. [P1] Hall StickyCartBar still ignores visit lifecycle state ‚Äî `showCartButton` and `hallStickyMode` only branch on `isTableVerified`, cart length, `myOrders.length`, and `sessionOrders.length`. They never inspect visit/session/bill/closed/paid state, so the bar cannot implement the closed/paid/reset behaviors documented in `BUGS.md` and the batch brief. FIX: derive a real hall sticky state from table session plus bill/payment/visit state and render `StickyCartBar` from that state machine instead of raw counts.
4. [P2] Hall StickyCartBar copy is still a 4-state shortcut ‚Äî `hallStickyButtonLabel` and `hallStickyModeLabel` only produce a small set of generic labels. The separate draft-mode vs visit-mode CTA/text matrix is still missing, so guests get ambiguous copy in several hall states. FIX: add a dedicated label resolver for the full hall sticky state matrix and stop deriving button text from the current 4-branch ternaries.
5. [P2] StickyCartBar animations cannot fire from this page state ‚Äî `x.jsx` passes only static counts/totals into `StickyCartBar` and does not track previous cart size or first-item transition anywhere, so the ‚Äúcount bump‚Äù and ‚Äúsoft rise on first item‚Äù behaviors from the context cannot be triggered from the parent state. FIX: track previous cart count / first-item transitions in `x.jsx` and pass explicit animation triggers or state flags to `StickyCartBar`.
6. [P2] Mode-switch removal toast has no local i18n fallback ‚Äî `handleModeChange()` calls `t('cart.items_removed_mode_switch', { count: removed })`, but `cart.items_removed_mode_switch` is missing from `I18N_FALLBACKS`. If the backend translation is absent for the active language, `makeSafeT()` returns an empty string and the guest gets a blank toast. FIX: add `cart.items_removed_mode_switch` to `I18N_FALLBACKS` or switch this call to `tr()` with an explicit fallback string.
