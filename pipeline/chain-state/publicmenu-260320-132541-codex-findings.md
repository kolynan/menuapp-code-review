# Codex Writer Findings - PublicMenu Chain: publicmenu-260320-132541
## Findings
1. [P0] Loyalty points are deducted before order creation - In `x.jsx:2444-2457` and `x.jsx:2818-2831`, the redeem transaction and balance update run before `Order.create()` at `x.jsx:2487` and `x.jsx:2854`. If order creation fails, the catch only sets `submitError`, so points are lost without an order. FIX: create the order first, then write the redeem transaction and balance update, or add a compensating rollback in the catch path.
2. [P1] Loyalty section crashes when `customerEmail` is null - `CartView.jsx:912` and `CartView.jsx:976` call `customerEmail.trim()` directly, while the input binds `value={customerEmail}` at `CartView.jsx:906`. If the loyalty hook returns `null` or `undefined`, CartView can throw during render. FIX: normalize once with `const normalizedCustomerEmail = (customerEmail ?? '').trim()` and pass `customerEmail ?? ''` to the input.
3. [P1] Bill cooldown check can crash in restricted browsers - `x.jsx:283-287` reads `localStorage` without try/catch, and the effect at `x.jsx:2258-2260` calls it on mount. In private mode or blocked-storage environments, this can crash the page before the cart opens. FIX: wrap the storage access in try/catch and return `false` on failure.
4. [P2] Reward-email mini-form shows false success and accepts invalid emails - `CartView.jsx:524-535` accepts any non-empty string, calls `setCustomerEmail(rewardEmail)`, shows a success toast immediately, and closes after 1s without validating format or waiting for loyalty lookup/persistence. Guests can see "saved" even when the email is malformed or lookup fails. FIX: validate the email, await the real save/lookup result, and show an error toast on failure.
5. [P2] Submit-error subtitle falsely says the order was saved - `CartView.jsx:1227-1228` always renders `error.send.subtitle`, and `x.jsx:403` defines it as "Your order is saved. Try again." even when `Order.create()` failed completely. This misleads guests after a failed submit. FIX: change the copy to neutral retry text, or only show the "saved" variant when there is explicit persistence confirmation.
6. [P2] Locale and currency formatting are hardcoded to Russian/Tenge - `CartView.jsx:409`, `CartView.jsx:929-970`, `CartView.jsx:1000`, `CartView.jsx:1044`, `x.jsx:973`, and `x.jsx:1206` hardcode `ru-RU`, `ru-RU` time formatting, a Tenge-only symbol, and a raw `B` points suffix instead of using the active language and formatter. Non-Russian partners get wrong separators, time format, and currency text. FIX: reuse the page locale and `formatPrice`, and move point/unit labels through i18n keys.
7. [P2] Zero redeem-rate partners display the wrong money equivalent - `CartView.jsx:932` uses `(partner?.loyalty_redeem_rate || 1)`, so an explicit rate of `0` is treated as `1`. Guests see redeemable value that does not exist. FIX: replace `|| 1` with `?? 1`.
8. [P2] Redirect-banner timer leaks after unmount or effect reruns - `x.jsx:1866-1871` starts `setTimeout(() => setRedirectBanner(null), 5000)` with no cleanup. If the page unmounts or the effect reruns first, stale timers still call state updates. FIX: store the timer id and clear it in the effect cleanup.
9. [P2] Table-code UI overflows narrow phones for long codes - `CartView.jsx:103-106` allows up to 8 digits, but the verifier row at `CartView.jsx:1085-1092` renders fixed `w-9` boxes with `gap-2`. Eight boxes exceed a 320px viewport once padding is included, so the code row clips or squeezes. FIX: make the box width and gap responsive, or switch to a single-input or wrapped layout for longer codes.
10. [P2] Production debug logging exposes guest and order data - `x.jsx:2269-2306` dumps guest ids, names, orders, and item mappings to `console.log` behind a query param, and `x.jsx:2591` / `x.jsx:2919` always log created order ids. This leaves debug instrumentation in guest-facing code and leaks operational data to the browser console. FIX: remove the logs or gate them behind a build-time dev flag that never ships to production.
11. [P3] Several icon-only controls are not mobile-accessible - `CartView.jsx:431-437`, `CartView.jsx:459-460`, and `CartView.jsx:1018-1024` use icon-only buttons with `title` or no label, and some hit areas are below the 44x44 mobile target. Screen-reader users do not get a usable name, and touch accuracy is poor. FIX: add `aria-label`s and enforce 44px minimum touch targets for the bell, save, cancel, and info buttons.
