## Questions Analyzed

### Q1: What 2 improvements should be made to `cleanup_stale_git_locks`?

**Recommendation:** Two improvements targeting safety and observability.

#### Improvement 1: Add a `--dry-run` / preview mode and log lock file age before deletion

**Reasoning:** Currently the helper silently deletes any `.lock` file older than `stale_age_sec`. When debugging lock-related issues, operators have no visibility into *how stale* a lock was or whether it was genuinely orphaned vs. held by a slow `git` process that was still running. The function should:
- Log each lock file's age (in seconds) and owning PID (if recoverable from the lock content) before deleting.
- Accept an optional `dry_run=False` parameter that logs what *would* be deleted without actually removing anything.

This makes the function safe to call speculatively (e.g., in a health-check loop) and gives better post-mortem data when a lock deletion causes a downstream git failure.

**Trade-offs:** Slightly more complex signature; dry-run param adds a code path that must be tested. Minimal runtime cost (one extra `stat` call per lock file, which is already being done for mtime).

**Mobile UX:** N/A (infrastructure function).

#### Improvement 2: Guard against race conditions with retry-after-delete verification

**Reasoning:** There is a TOCTOU (time-of-check-time-of-use) window: between checking `mtime > stale_age_sec` and calling `os.remove()`, a new git process could create a *fresh* lock at the same path. Deleting that fresh lock would corrupt an in-progress git operation. The function should:
- After `os.remove()`, immediately check if the file reappeared (within ~50ms). If it did, a live git process is actively using it — log a warning and do NOT re-delete.
- Optionally, before deleting, check if a `git` process is running that targets the same repo (e.g., `subprocess.run(["git", "-C", repo_dir, "status"], timeout=2)` — if it hangs, a real lock holder exists).

This turns a "best-effort cleanup" into a safe cleanup that won't stomp on concurrent git operations — important because the function is called from two hot paths (pre-flight in `expand_chain_task_if_needed` and pre-stash in `merge_worktree_to_main`), both of which may race with other worktree tasks.

**Trade-offs:** Adds ~100ms latency per lock file (for the reappearance check or process check). In the common case (no locks exist) there is zero overhead. The subprocess approach (`git status`) could itself hang if the lock is truly stuck, so it needs a short timeout.

## Additional Notes
- The function currently targets 5 specific lock paths (`index.lock`, `ORIG_HEAD.lock`, `HEAD.lock`, `MERGE_HEAD.lock`, `refs/heads/main.lock`). A minor third improvement (not requested) would be to glob `refs/heads/*.lock` to cover feature branches, but this risks deleting locks for active rebase/merge on non-main branches — so the hardcoded list is actually safer for now.
- The `stale_age_sec=60` default is reasonable for CI-like environments but may be too aggressive for slow network filesystems. The dry-run mode from Improvement 1 would help operators tune this value.
