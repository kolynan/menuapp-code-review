# Codex Reviewer Findings — ПССК Prompt Quality Review
Chain: publicmenu-260407-104609-7148

## Issues Found
1. [CRITICAL] Entity API is internally inconsistent — The prompt tells the implementer to grep for `Order.create(orderData)` / `OrderItem.bulkCreate`, but the replacement snippet switches to `base44.entities.Order.create(...)` / `base44.entities.OrderItem.create(...)`. If `base44` is not the current in-scope API in `x.jsx`, a literal implementation will fail or mix patterns inside one handler. PROMPT FIX: say "keep the existing entity access pattern already used in `x.jsx`; replace only the 1-order + bulkCreate flow with a loop." Remove hard-coded `base44.entities.*` unless confirmed by the current file.
2. [CRITICAL] "One order per cart item" does not guarantee per-dish control — The business goal is per-dish fulfillment, but the prompt loops over `cart` entries and preserves `quantity: item.quantity`. If a cart line has quantity 2 or 3 of the same dish, this still creates one order containing multiple units, so SOM may still advance multiple physical dishes together. PROMPT FIX: explicitly define the required behavior for `quantity > 1`. Either explode line items into unit orders (`quantity: 1` repeated N times) or change the acceptance criteria to say control is per cart line, not per dish.
3. [CRITICAL] Shared charges and item metadata are under-specified — `total_amount: itemTotal` is computed only from `price * quantity - discount`. That ignores delivery fees, service fees, packaging, tips, and similar shared charges. The sample `OrderItem.create(...)` also only includes a small field subset and may drop modifiers, add-ons, notes, or variant data carried by the current `bulkCreate` mapping. PROMPT FIX: instruct the implementer to preserve every existing `Order` and `OrderItem` field from the current code, and explicitly define how any non-item charges are allocated in delivery/takeaway mode.
4. [CRITICAL] Rollback is not fully defined and assumes cascade behavior — The prompt says to delete created orders on failure, but does not say whether deleting an `Order` automatically deletes its `OrderItem`. It also does not define what happens if failure occurs after the loop during loyalty attach, partner counter update, confirmation setup, or cart clear. That can leave partial data or inconsistent side effects. PROMPT FIX: define the atomic boundary. State whether `Order.delete` cascades to `OrderItem`; if not, delete created items explicitly. Also specify whether post-loop failures must trigger rollback of orders/items and how loyalty/counter side effects are compensated.
5. [MEDIUM] `splitType ?? 'single'` is not a safe fallback if `splitType` is undeclared — The prompt later admits `splitType` may not be available in one branch, but the sample code still references it directly. In JavaScript, `splitType ?? 'single'` throws if `splitType` was never declared. PROMPT FIX: replace the sample with a safe rule such as "preserve the current branch's field mapping; if no `splitType` variable exists there, use the literal `'single'`."
6. [MEDIUM] Scope lock conflicts with required changes — The prompt says "modify ONLY the two order-creation blocks" and "do NOT change any state," but later requires changing `showConfirmation(...)`, `setSessionOrders`, and `setSessionItems`. Those are likely adjacent but not necessarily inside the narrow create blocks. PROMPT FIX: redefine scope as "changes are limited to `handleSubmitOrder` logic in `pages/PublicMenu/x.jsx`, including post-submit optimistic updates and confirmation token wiring."
7. [MEDIUM] Confirmation token strategy is ambiguous — The prompt requires unique `public_token` per split order but then says to use only `createdOrders[0].public_token` for confirmation. If confirmation or later reloads fetch by `public_token`, only one order may be recoverable after refresh, even if optimistic state initially looks correct. PROMPT FIX: state exactly how confirmation/cart reload should resolve the full split submission after a hard refresh, and whether grouping is by guest, session, table, or another stable key.
8. [MEDIUM] Discount rules are incomplete for zero/free/full-discount edge cases — The sample math divides by `cartSubtotal` and assumes it is positive. It also does not say how to clamp negative `itemTotal` values when discounts equal or exceed subtotal, or how to handle free items mixed with paid items. PROMPT FIX: add explicit rules for `cartSubtotal <= 0`, free items, full-cart discounts, and negative-total prevention. Define rounding and clamping behavior unambiguously.
9. [MEDIUM] Side-effect semantics are not defined for partner counters and loyalty — "Call `Partner.update(updatedCounters)` once, not N times" and "attach loyalty to `createdOrders[0]` only" may be correct, but the prompt never states whether those systems are meant to count cart submissions or actual orders. After the split, analytics, daily counters, rewards, refunds, and staff metrics may drift. PROMPT FIX: specify the intended business semantics for counters and loyalty after splitting, including whether counts should increment by 1 submission or by N persisted orders.
10. [MEDIUM] Verification is too UI-heavy and does not prove persisted correctness — Most checks are manual UI checks and can pass from optimistic state even if persisted orders are wrong. There is no explicit check for hard refresh, duplicate-submit behavior with slower sequential creates, orphan records after rollback, or branch-specific shared fields (`guest`, `table`, `table_session`, address) surviving on every created order. PROMPT FIX: add verification for hard refresh/reload, double-tap/submit-lock behavior, rollback cleanup, and persistence of shared fields in both hall and delivery/takeaway branches.
11. [LOW] Search guidance is brittle for the stated environment — The prompt uses `grep -n ...` commands and fixed approximate line numbers. In a Windows/PowerShell workflow, `grep` may not exist, and stale line offsets encourage brittle navigation. PROMPT FIX: use shell-neutral guidance or `rg` examples, and label line numbers as approximate hints only, not expected locations.
