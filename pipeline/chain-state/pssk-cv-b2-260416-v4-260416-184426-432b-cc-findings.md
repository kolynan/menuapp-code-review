### MEDIUM

1. **[MEDIUM] Fix 3 storage strategy diverges from R4 spec (single key vs per-table keys)** — R4 in FROZEN UX table says "Durable persist `cv_terminal_dismissed_{tableId}` (localStorage)" — template implies **per-table keys** (e.g. `cv_terminal_dismissed_12`, `cv_terminal_dismissed_34`). But Шаг 3.1/3.3 use a **single key** `cv_terminal_dismissed` that stores the last dismissed tableId as its value. Behavioral diff: if guest dismisses at Table A, then visits Table B (also closed) and dismisses → storage now holds "B" → if guest returns to Table A, condition `stored("B") !== "A"` becomes true → **terminal re-appears for Table A even though it was already dismissed**. Per-table keys would have remembered A's dismissal permanently. PROMPT FIX: Either (a) update implementation to `localStorage.setItem(\`cv_terminal_dismissed_${currentTableKey}\`, '1')` + read via template key on mount/when currentTableKey changes (via useEffect), OR (b) explicitly note in prompt "по решению — используем один глобальный ключ с tableId-значением; поведение ≠ per-table спеке" to acknowledge the divergence.
