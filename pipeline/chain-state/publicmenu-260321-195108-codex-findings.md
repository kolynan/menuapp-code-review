# Codex Writer Findings ‚Äî PublicMenu
Chain: publicmenu-260321-195108

## Findings
1. [P1] Hall submit validates before the Bottom Sheet gate ‚Äî In `pages/PublicMenu/base/x.jsx`, `handleSubmitOrder()` calls `validate()` before the just-in-time table-confirmation check. `validate()` rejects hall submits when `!currentTableId`, but that error is stored as `errors.tableSelection` and is not rendered anywhere in the current drawer flow. Result: on `/x?partner=...&mode=hall` a new guest can tap "–û—Ç–ø—Ä–∞–≤–∏—Ç—å –æ—Ñ–∏—Ü–∏–∞–Ω—Ç—É" and get a silent no-op instead of the confirmation sheet. FIX: move the `orderMode === "hall" && !isTableVerified` interception before `validate()`, or skip `tableSelection` validation until after the sheet verifies the table.
2. [P1] Pickup/delivery checkout still drops loyalty and discount UI ‚Äî `pages/PublicMenu/base/x.jsx` passes `showLoyaltySection`, `customerEmail`, `loyaltyAccount`, `earnedPoints`, `redeemedPoints`, `maxRedeemPoints`, `discountAmount`, and related props into `CheckoutView`, but `pages/PublicMenu/base/CheckoutView.jsx` neither accepts nor renders them. Users in pickup/delivery lose the email, points, redeem, and discount flows on the actual checkout screen. FIX: add the missing props to `CheckoutView` and render the loyalty block there, or extract the existing loyalty UI into a shared checkout section.
3. [P2] Table-confirmation state lives in hidden `CartView` instead of the visible Bottom Sheet ‚Äî After inline verification UI was removed, `pages/PublicMenu/base/CartView.jsx` still owns `codeAttempts`, `codeLockedUntil`, `codeSecondsLeft`, and the auto-verify effect bound to `tableCodeInput`, while the visible input now lives in the Bottom Sheet in `pages/PublicMenu/base/x.jsx`. That means the sheet shows only `codeVerificationError` with no attempt/cooldown feedback, and verification can be triggered by an off-screen component before the CTA tap. FIX: move attempt/lockout/auto-verify state into `x.jsx` with the Bottom Sheet, or remove the hidden auto-verify effect and let the sheet CTA own verification.
4. [P2] Mobile grid partner setting is ignored ‚Äî `pages/PublicMenu/base/MenuView.jsx` computes `mobileCols` from `partner.menu_grid_mobile`, but the actual grid render hardcodes `isMobile ? 'grid-cols-2' : gridColsClass`. Any partner configured for a 1-column mobile menu still gets 2 columns. FIX: use the computed mobile class (`MOBILE_GRID[mobileCols]`) in the mobile branch instead of the hardcoded `grid-cols-2`.
5. [P2] `useTableSession` loses restored guests that only have `_id` ‚Äî The hook correctly defines `normalizeGuestId()`, but later syncs `currentGuestIdRef.current = currentGuest?.id || null` in `pages/PublicMenu/base/useTableSession.jsx`. If a restored `SessionGuest` only carries `_id`, the ref is reset to `null`, so polling keeps re-running guest restore and downstream logic can behave as if no current guest exists. FIX: store `normalizeGuestId(currentGuest)` in the ref, and align any other guest-id reads in this file with the same normalization rule.
6. [P3] Several control labels still regress i18n/accessibility ‚Äî `pages/PublicMenu/base/CheckoutView.jsx` hardcodes `aria-label="Decrease"` / `aria-label="Increase"`, `pages/PublicMenu/base/CartView.jsx` hardcodes the reward-email placeholder `email@example.com`, and `pages/PublicMenu/base/MenuView.jsx` list-mode add/remove buttons plus `CartView` icon-only controls are missing localized accessible labels. This leaves mixed-language assistive text and unlabeled mobile controls in production. FIX: replace hardcoded assistive/user-facing strings with `t()/tr()` and add `aria-label` to icon-only/list-stepper buttons.
