# Codex Writer Findings - PublicMenu
Chain: publicmenu-260320-141634

## Findings
1. [P1] Partial order commits are reported as full failure - In both submit paths, `Order.create()` happens before several other awaited writes, but the whole sequence still shares one catch block (`x.jsx:2477-2535,2597-2600` and `x.jsx:2844-2892,2924-2929`). If `LoyaltyTransaction.create`, `LoyaltyAccount.update`, `OrderItem.bulkCreate`, or `Partner.update` fails after the order is created, the UI stays in retry state even though an order may already exist, which can produce duplicate or partial orders on resubmit. FIX: track the post-create phase explicitly; once order plus items are persisted, treat later side effects as best-effort, or add compensating rollback/cancel logic for already-created orders/items before surfacing a retry UI.
2. [P2] Accepted statuses still collapse to "new" and badges ignore icons - `getSafeStatus()` still has no fallback for `accepted` (`CartView.jsx:243-262`), and the table-order / my-order badges render only `status.label` instead of `{status.icon} {status.label}` (`CartView.jsx:603,611,694`). Guests can lose the distinction between accepted, cooking, and ready states when a raw translation key reaches the UI. FIX: add an `accepted` fallback and render the icon in every status badge location.
3. [P2] Drawer scroll is not reset after successful table verification - The `isTableVerified` effect now only clears attempt/cooldown state (`CartView.jsx:143-148`). If the guest enters the code at the bottom of the drawer, the verified cart remains stranded low on screen instead of returning to the top. FIX: restore the false->true verification transition handler that scrolls the nearest scrollable drawer container back to top.
4. [P2] Guest code is leaked into the main header label - `effectiveGuestCode` falls back to `menu_guest_code` from localStorage and `guestDisplay` always appends `#code` (`CartView.jsx:272-293`). That exposes an internal waiter/session code in the "You:" label even when `hallGuestCodeEnabled` is off. FIX: gate storage fallback behind `hallGuestCodeEnabled` and show the code only inside the dedicated waiter-code block.
5. [P2] Review-reward CTA invents a +10 bonus when no reward exists - The "rate guests' dishes" button appends a points bonus whenever `loyaltyAccount` exists, using `(partner?.loyalty_review_points || 10)` (`CartView.jsx:626-635`). If the partner disabled review rewards or set them to `0`, the UI still promises `+10`. FIX: show bonus text only when review rewards are active and replace `|| 10` with `?? 0`.
6. [P2] Reward-email flow accepts invalid input and shows success before anything succeeds - The save handler accepts any non-empty string, calls `setCustomerEmail`, immediately shows a success toast, then auto-closes after 1s (`CartView.jsx:514-539`). Invalid emails and failed downstream loyalty lookup/persistence still look successful; the placeholder is also hardcoded English. FIX: validate email format first, await the real save/lookup result, show an error toast on failure, and localize the placeholder text.
7. [P2] Formatting still bypasses the active app locale - CartView formats loyalty points with raw `toLocaleString()` and inline text (`CartView.jsx:409,933,969-977,1005,1049`), while `OrderStatusScreen` formats totals and time with browser-default locale logic (`x.jsx:971-978,1205-1210`). On devices whose browser locale differs from the selected app language, the page shows inconsistent separators/time formatting. FIX: pass one shared locale-aware formatter into CartView and `OrderStatusScreen` and use it for points, money, and times.
8. [P2] Verified-state online-order panel still wastes mobile space - After table verification, the full amber "online order to waiter" block still renders even when there are no discounts, bonus earnings, or point savings to explain (`CartView.jsx:1013-1060`). On a small drawer this pushes the primary CTA lower and duplicates the state with a second "table verified" message. FIX: restore a `shouldShowOnlineOrderBlock` condition so the panel hides after successful verification unless it has actual benefit content.
9. [P2] Long table-code layouts overflow narrow phones - The code-slot row uses fixed `w-9` boxes and fixed gaps while allowing up to 8 digits (`CartView.jsx:1079-1093`). On 320px-class screens the slot row exceeds the viewport width and clips the verification UI. FIX: make slot width/gaps responsive or switch long codes to a wrapped/single-input layout.
10. [P2] Production debug mode still exposes guest/order data in the console - Visiting `?debug=guestItems` logs guest IDs, names, guest numbers, per-guest item maps, and order associations (`x.jsx:2268-2311`). A query-param gate is still production code, not a safe dev-only guard. FIX: remove these logs or protect them behind a build-time dev flag that never ships.
11. [P3] Several icon-only controls are unlabeled and too small for touch - The waiter bell, inline name save/cancel glyphs, drawer close button, remove-item `X`, and info buttons rely on `title` or no label at all, and several render below a 44x44 target (`CartView.jsx:431-437,459-460,474-480,788-793,1018-1024,1067-1074`). This hurts screen-reader use and mobile tapping. FIX: add explicit `aria-label`s and enforce a minimum 44px hit area for every icon-only button.
