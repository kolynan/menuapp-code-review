# Codex Writer Findings — PublicMenu
Chain: publicmenu-260322-114012

## Findings
1. [P1] Dynamic primary color is still hardcoded in `x.jsx` — `x.jsx` never defines `primaryColor` / `darkenColor` / `lightenColor`, and terracotta `#B5543A` is still baked into live UI at `x.jsx:751`, `x.jsx:770`, `x.jsx:1024`, `x.jsx:1156-1157`, `x.jsx:3032`, `x.jsx:3190`, `x.jsx:3426`, and `x.jsx:3460`. Partners with a custom `primary_color` still see terracotta in confirmation CTAs, order-status UI, page loading, active category chips, and the table-confirm sheet. FIX: add module-scope color helpers, compute `const primaryColor = partner?.primary_color || '#1A1A1A'` inside `X()`, replace all hardcoded terracotta usages, pass `primaryColor` to child components, and replace the static `focus-within:border-[#B5543A]` with state-driven inline border styling.
2. [P1] Partner lookup hides backend failures as "restaurant not found" — the partner query catches both primary and fallback lookup errors and returns `null` (`x.jsx:1322-1345`), then the page always renders `t('error.partner_not_found')` when `partner` is falsy (`x.jsx:3037-3044`). Temporary API failures are misreported as a bad QR link, and React Query cannot retry because the query resolves successfully with `null`. FIX: preserve request failures separately from a true empty lookup; only show `error.partner_not_found` for an actual miss and render a retryable backend/network error state for thrown errors.
3. [P1] Order tracking also misclassifies fetch failures as "not found" — `OrderStatusScreen` renders the same not-found screen for both `orderError` and a genuinely missing order (`x.jsx:895-905`, `x.jsx:1030-1045`). A transient API failure makes an existing order look invalid, blocking recovery from the tracking screen. FIX: split the branch so `orderError` shows a retryable error state, while `order_status.not_found` is reserved for a successful query that returns no order.
4. [P1] Hall StickyCartBar still uses a 4-state heuristic instead of the visit lifecycle — `hallStickyMode` only chooses among `cart`, `myBill`, `tableOrders`, and `cartEmpty`, and all button/label logic is derived from that same simplified state (`x.jsx:2101-2144`). This ignores the visit/draft/paid/closed matrix already documented in `BUGS.md`, so the sticky bar shows the wrong CTA/copy/visibility after order submission, after bill/closure, and when a visit exists without cart items. FIX: derive sticky-bar state from the full table-session lifecycle and map button text, secondary label, amount visibility, and hidden states from that complete matrix.
5. [P2] Table-confirm bottom sheet still hides retry/cooldown context — the sheet only renders `codeVerificationError` as a generic line of text (`x.jsx:3417-3452`) and does not show attempts remaining or lockout countdown, even though the fallback keys for attempts/locked state already exist in the file (`x.jsx:466-479`). On mobile this leaves the guest without actionable feedback during cooldown. FIX: lift verification attempt / lockout state into `x.jsx` and render attempts remaining plus remaining cooldown time inside the bottom sheet.
6. [P2] Auto-submit after table verification uses an uncleared timer — once verification succeeds, the effect closes the sheet and schedules `handleSubmitOrder()` with raw `setTimeout(..., 100)` (`x.jsx:2091-2097`). That timer is not stored or cleaned up, so a quick unmount/navigation can still fire a stale submit callback. FIX: store the timeout id in a ref, clear it on cleanup / sheet close / unmount, and call a stable submit callback from that timer.
7. [P3] Production `console.error` calls remain in user flows — error logs are still emitted from rating save, guest-code init, rate-limit check, loyalty side effects, order submit, guest rename, and bill request paths (`x.jsx:2066`, `x.jsx:2159`, `x.jsx:2408`, `x.jsx:2512`, `x.jsx:2543`, `x.jsx:2547`, `x.jsx:2889`, `x.jsx:2920`, `x.jsx:2950`, `x.jsx:2976`, `x.jsx:3007`). This leaks internal details to production consoles without improving guest-facing recovery. FIX: remove production `console.error` calls or route them through a debug-only logger while keeping the existing user-facing toasts/errors.
