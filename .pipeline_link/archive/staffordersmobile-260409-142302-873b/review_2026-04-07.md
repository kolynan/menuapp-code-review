# Code Review Report: staffordersmobile.jsx

## Summary
`pages/StaffOrdersMobile/staffordersmobile.jsx` still contains two P0 access-scope bugs, several P1 workflow regressions, and unresolved platform-rule violations. I reviewed the current file against `StaffOrdersMobile README.md`, `BUGS.md`, and the cleanup helper that this page calls.

## Findings

- [P0] Global session cleanup runs before access is validated - `staffordersmobile.jsx:2844-2858` calls `runSessionCleanup({ dryRun: false })` on mount for every visitor, before `gateView` blocks invalid or unauthorized states. The helper then fetches all open sessions and updates them globally in `components/sessionCleanupJob.js:147-198`. A locked token, inactive invite, or otherwise unauthorized visit can still expire sessions belonging to other restaurants. FIX: only start cleanup after access is validated and the current restaurant is known, and scope the cleanup helper by `partnerId` before it reads or mutates sessions.

- [P0] Broad cross-restaurant data reads happen before partner access is confirmed - `canFetch` becomes true for any logged-in user (`staffordersmobile.jsx:3216-3219`), while the access UI check only happens later via `gateView` (`staffordersmobile.jsx:3141-3208`). When `partnerId` is null, the page falls back to `Table.list()`, `Order.list()`, and `ServiceRequest.list()` (`staffordersmobile.jsx:3234-3238`, `staffordersmobile.jsx:3359-3366`, `staffordersmobile.jsx:3381-3388`). Invalid-role or no-partner users can therefore hit broad entity reads before the page tells them "ąØąĄčé ą┤ąŠčüčéčāą┐ą░". FIX: require validated access plus a non-null `partnerId` before enabling any data queries, and remove the `*.list()` fallbacks from this page.

- [P1] Shared `updateLinkMutation.isPending` can blank the token UI during unrelated writes - token-mode `gateView` treats `updateLinkMutation.isPending` as a binding state (`staffordersmobile.jsx:3163-3166`), but the same mutation object is also used for favorite-table writes and other `StaffAccessLink` updates (`staffordersmobile.jsx:2983-2987`, `staffordersmobile.jsx:3002-3005`, `staffordersmobile.jsx:3127-3131`). On a slow network, tapping a favorite can replace the whole screen with `BindingScreen`. FIX: split device binding/unbinding into a dedicated mutation or guard the screen with a separate `isBinding` flag instead of the shared mutation pending state.

- [P1] Auto-bind can dead-end permanently on ordinary API errors - the bind effect sets `didBindRef.current = true` before calling `updateLinkMutation.mutate(...)` (`staffordersmobile.jsx:3038-3045`), but `updateLinkMutation.onError` only handles rate limits and never resets that guard (`staffordersmobile.jsx:2952-2969`). If the bind request fails normally, token users stay in `bind_needed` state forever with no retry path. FIX: reset the bind guard on non-429 failures and show an explicit retry/error state for binding failures.

- [P1] Logout still races device unbinding - `handleLogout` fires `updateLinkMutation.mutate(...)`, immediately clears local data, then redirects after 500 ms (`staffordersmobile.jsx:3126-3138`). If the server request is slow or fails, `bound_device_id` can remain on the old device and the next login will be locked out. FIX: make logout async, await the unbind request, and only redirect after the server-side state is updated or the user explicitly chooses to continue without unbinding.

- [P1] Standalone service requests still use the obsolete lifecycle and drop ownership metadata - `RequestCard` still renders `new -> in_progress -> done` labels (`staffordersmobile.jsx:1486-1488`), and the standalone request list still mutates `new ? "in_progress" : "done"` (`staffordersmobile.jsx:4210-4217`). That contradicts the current `BUGS.md` flow, where grouped table requests already use `accepted -> done` plus `assignee` / `assigned_at` (`staffordersmobile.jsx:2234`). FIX: reuse the same accepted/done request action builder for standalone requests and include `assignee` plus `assigned_at` when a waiter accepts a request.

- [P1] Kitchen filtering leaks first-stage orders for custom stage pipelines - the kitchen-only filter checks `stage.internal_code !== 'start'` (`staffordersmobile.jsx:3455-3467`) instead of reusing the existing first-stage detection logic from `getStatusConfig`. If a restaurant uses a custom first stage that is first by order but not literally `internal_code === 'start'`, kitchen users will see orders that should still be waiter-only. FIX: base the kitchen filter on `getStatusConfig(order).isFirstStage` or the first relevant sorted stage, not only on `internal_code`.

- [P1] The served section can show historical orders from previous table sessions - `servedOrders` is loaded with `Order.filter({ table: group.id, status: "served" })` (`staffordersmobile.jsx:1749-1754`). That query ignores `table_session`, so a current table card can include served orders from an earlier session on the same table. FIX: scope the served-orders query to the current `table_session` (or at minimum the current shift/session window) instead of only the table id.

- [P1] Link/user query failures are rendered as auth/access problems instead of retriable errors - `linkError` and `userError` only trigger special handling for rate limits (`staffordersmobile.jsx:2872-2893`). Ordinary fetch failures then fall through into misleading UI such as "ąĪčüčŗą╗ą║ą░ ąĮąĄą┤ąĄą╣čüčéą▓ąĖčéąĄą╗čīąĮą░", "ąØąĄąŠą▒čģąŠą┤ąĖą╝ąŠ ą▓ąŠą╣čéąĖ", or "ąØąĄčé ą┤ąŠčüčéčāą┐ą░" (`staffordersmobile.jsx:3167-3203`). FIX: add explicit non-rate-limit error states with retry actions for the link and current-user queries instead of collapsing them into auth/access messages.

- [P1] Order and request mutations fail silently outside 429 cases - `OrderCard` and `OrderGroupCard` optimistic mutations only rollback cache in `onError` (`staffordersmobile.jsx:1549-1562`, `staffordersmobile.jsx:1855-1870`), and `updateRequestMutation` only reacts to 429s (`staffordersmobile.jsx:3414-3423`). On normal API failures, the waiter gets no toast, no retry, and no explanation. FIX: add generic error handling for all mutation paths, surface a user-visible failure message, and provide a retry/recovery path.

- [P1] File-wide i18n is still missing - `useI18n()` is effectively only used to translate `OrderStage.name` (`staffordersmobile.jsx:278-295`, `staffordersmobile.jsx:2740`), while the rest of the page is still hardcoded Russian text: status/type constants (`staffordersmobile.jsx:213-482`), gate screens (`staffordersmobile.jsx:3167-3203`), profile/settings copy (`staffordersmobile.jsx:2376-2611`), toasts (`staffordersmobile.jsx:2782-2795`, `staffordersmobile.jsx:3009`, `staffordersmobile.jsx:3116`, `staffordersmobile.jsx:3135`, `staffordersmobile.jsx:3994-4041`), and the main screen (`staffordersmobile.jsx:4078-4405`). FIX: move all user-facing strings, fallbacks, placeholders, and aria labels to `t('staff_orders_mobile...')` keys and pass `t` into helper factories/constants.

- [P2] Shift cutoff never rolls forward while the page stays open - `shiftStartTime` is memoized only from `partnerData?.working_hours` (`staffordersmobile.jsx:3230-3232`), then reused by order/request filters (`staffordersmobile.jsx:3399-3412`, `staffordersmobile.jsx:3427-3451`). If a waiter keeps the page open across midnight or a shift boundary, the filter window stays stale until a full reload. FIX: recompute the shift cutoff from a live clock tick or derive it inline from current time instead of memoizing it only on working-hours changes.

- [P2] `parseTime` no longer guards against `NaN`, so malformed working hours can corrupt filtering - `getShiftStartTime()` parses `HH:MM` with `const [h, m] = timeStr.split(':').map(Number); return h * 60 + m;` (`staffordersmobile.jsx:821-825`). Invalid values flow into `setHours(...)` later (`staffordersmobile.jsx:849-865`), producing an invalid cutoff date and breaking the shift filter. FIX: restore a numeric guard in `parseTime` and treat malformed times as `null`, not as arithmetic input.

- [P2] AudioContext is leaked on unmount - `createBeep()` creates a new `AudioContext` (`staffordersmobile.jsx:993-1019`), but the page never closes it after mount (`staffordersmobile.jsx:2826-2840`). Reopening this page repeatedly can leave browser audio resources hanging around. FIX: add an unmount cleanup effect that calls `audioRef.current?.ctx?.close()` safely.

- [P2] Double `ąĪčéąŠą╗` prefix regression is back in live UI paths - the project docs already note that DB table names already include the prefix, but the current UI still prepends `"ąĪčéąŠą╗ "` in `OrderCard` (`staffordersmobile.jsx:1589-1593`) and `MyTablesModal` (`staffordersmobile.jsx:2332`). Users can see labels like `ąĪčéąŠą╗ ąĪčéąŠą╗ 5` again. FIX: route table labels through the existing guarded helper pattern instead of manual string concatenation.

- [P2] "My Tables" mixes table favorites and request favorites, then clears both under a table-only label - request favorites are stored in the same `favorites` array via `toggleFavorite('request', ...)` (`staffordersmobile.jsx:1497-1499`), but `MyTablesModal` shows `favorites.length` as the table count and `clearAllFavorites()` wipes the mixed array while saying "ąĪčéąŠą╗čŗ ąŠčćąĖčēąĄąĮčŗ" (`staffordersmobile.jsx:2290-2291`, `staffordersmobile.jsx:2343-2352`, `staffordersmobile.jsx:2998-3009`). FIX: separate table favorites from request favorites, or at minimum count/filter only `table:*` entries inside the modal and its clear action.

- [P2] Main waiter actions are still too small for a mobile-first screen - several live controls remain at `h-7` or `min-h-[36px]`, including the standalone request action button (`staffordersmobile.jsx:1513`), request accept/serve buttons in expanded hall cards (`staffordersmobile.jsx:2234`), and multiple batch buttons in hall/legacy sections (`staffordersmobile.jsx:616`, `staffordersmobile.jsx:648`, `staffordersmobile.jsx:671`, `staffordersmobile.jsx:686`, `staffordersmobile.jsx:2250-2252`). These are primary operational controls on a phone and still fall below the 44 px touch-target floor. FIX: raise all primary action controls in the order/request flows to at least `min-h-[44px]` with consistent padding.

- [P2] Production debug logging is still in the page - session cleanup writes to `console.log` on both success and error paths (`staffordersmobile.jsx:2848-2852`). This is a staff-facing production screen and the repo rules explicitly forbid debug logs. FIX: remove these logs or route them through a controlled telemetry layer outside the page component.

- [P3] The file still carries large dead/commented code blocks that make regressions easier - the header changelog (`staffordersmobile.jsx:1-157`) and the commented-out legacy JSX tree (`staffordersmobile.jsx:523-1391`) leave multiple historical UI versions in one file. That materially increases review noise and the risk of fixing the wrong branch. FIX: move release history into markdown docs and delete dead commented JSX from the source file.
