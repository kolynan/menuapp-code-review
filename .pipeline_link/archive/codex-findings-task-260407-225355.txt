## Summary
`staffordersmobile.jsx` still has two P0 cross-tenant risks and several P1 workflow regressions. `BUGS.md` was useful context: the accepted/done request flow and the deferred i18n debt are both documented there, and the current file has regressed the request flow while never resolving the i18n problem.

## Confirmed Findings

- [P0] Global session cleanup runs before any access or partner gate - `staffordersmobile.jsx:2844` calls `runSessionCleanup({ dryRun: false })` on mount for every visitor. The helper then loads all open sessions in `components/sessionCleanupJob.js:147` and updates them globally in `components/sessionCleanupJob.js:194`. A locked token, inactive link, or otherwise unauthorized visit can still expire other restaurants' sessions. FIX: only run cleanup after access is validated for the current restaurant, pass the current `partnerId` into the helper, and scope `TableSession`/`OrderStage`/`Order` queries by partner before mutating anything.

- [P0] Access gating happens after broad data queries, so unauthorized users can fetch cross-restaurant data - `gateView` blocks UI only after hooks have already run (`staffordersmobile.jsx:3141`). Meanwhile `canFetch` becomes true for any logged-in user or token-mode link (`staffordersmobile.jsx:3216`), and when `partnerId` is null the page falls back to `Table.list()`, `Order.list()`, and `ServiceRequest.list()` (`staffordersmobile.jsx:3236`, `staffordersmobile.jsx:3361`, `staffordersmobile.jsx:3383`). Invalid-role / no-partner states can therefore hit broad entity reads before the page says "Нет доступа". FIX: require validated access plus a non-null normalized `partnerId` before enabling queries, and remove the `*.list()` fallbacks entirely from this page.

- [P1] Token auto-bind can dead-end on generic failures - the bind effect sets `didBindRef.current = true` before mutating (`staffordersmobile.jsx:3038-3045`), but `updateLinkMutation.onError` only handles 429s and never resets that guard on ordinary failures (`staffordersmobile.jsx:2952-2969`). Because `gateView` keeps returning `BindingScreen` while `tokenState === "bind_needed"` (`staffordersmobile.jsx:3163-3166`), a transient API/network failure can leave token users stuck with no retry path. FIX: track binding state separately, reset the bind guard on non-rate-limit failures, and show an explicit retry/error state.

- [P1] Any pending `StaffAccessLink` mutation can incorrectly flip the whole page into `BindingScreen` - `gateView` treats `updateLinkMutation.isPending` as equivalent to "currently binding" (`staffordersmobile.jsx:3165`), but the same mutation object is also used for favorites updates (`staffordersmobile.jsx:2983-2987`) and clear-all favorites (`staffordersmobile.jsx:3002-3005`). In token mode, starring or clearing favorites can blank the page behind the binding screen until the mutation finishes. FIX: split binding/unbinding into its own mutation or gate on a dedicated `isBinding` flag instead of the shared mutation pending state.

- [P1] Logout still races device unbinding - `handleLogout` starts `updateLinkMutation.mutate(...)` and then immediately clears local data and redirects after 500 ms (`staffordersmobile.jsx:3126-3138`). On a slow or failed request, `bound_device_id` can remain on the old device and lock the link for the next login. FIX: make logout async, await the unbind update, handle failure explicitly, and redirect only after the server-side unbind succeeds or the user chooses to continue anyway.

- [P1] Kitchen filtering leaks first-stage orders for custom stage pipelines - the kitchen filter uses only `stage.internal_code !== "start"` (`staffordersmobile.jsx:3455-3467`). Elsewhere the page already supports first-stage detection by actual stage position, not only by `internal_code`, so restaurants using a custom first stage without that exact internal code will expose new orders to kitchen users. FIX: reuse `getStatusConfig(order).isFirstStage` or compare against the first relevant sorted stage instead of relying only on `internal_code`.

- [P1] Standalone service requests still use the obsolete request lifecycle and drop assignee metadata - `BUGS.md:10-28` and `BUGS.md:47-50` document the move to `new -> accepted -> done`, but standalone `RequestCard` still renders `new -> in_progress -> done` labels (`staffordersmobile.jsx:1486-1488`) and the standalone list still mutates `new ? "in_progress" : "done"` (`staffordersmobile.jsx:4215`). Grouped table requests already use `accepted` plus `assignee` / `assigned_at` (`staffordersmobile.jsx:2234`). FIX: reuse the same accepted/done action builder for standalone requests and include `assignee` plus `assigned_at` on accept.

- [P1] Core order and request actions fail silently outside rate-limit cases - order mutations only rollback cache in `staffordersmobile.jsx:1558-1561` and `staffordersmobile.jsx:1864-1870`, while request mutations only react to 429s in `staffordersmobile.jsx:3417-3422`. On normal API failures, waiters get no toast, no retry path, and no actionable error state, so taps appear to do nothing. FIX: add generic error handling for all action mutations, surface a user-visible failure message, and provide a retry/recovery path.

- [P1] File-wide i18n violation - `useI18n()` is only used to translate `OrderStage.name` (`staffordersmobile.jsx:278-295`, `staffordersmobile.jsx:2740`), while almost all visible UI remains hardcoded Russian text: status/type constants (`staffordersmobile.jsx:213-349`), gate screens (`staffordersmobile.jsx:1393-1437`, `staffordersmobile.jsx:3167-3203`), profile/settings sheets (`staffordersmobile.jsx:2376-2611`), toasts (`staffordersmobile.jsx:2782-2795`, `staffordersmobile.jsx:3009`, `staffordersmobile.jsx:3116`, `staffordersmobile.jsx:3135`, `staffordersmobile.jsx:4037-4041`), and main page copy (`staffordersmobile.jsx:4078-4243`). `BUGS.md:361-365` already tracks this as an active deferred bug. - [P2] Shift cutoff is memoized once and never rolls forward while the page stays open - `shiftStartTime` is calculated with `useMemo` only from `partnerData?.working_hours` (`staffordersmobile.jsx:3230-3232`), but that cutoff is then reused by both request and order filters (`staffordersmobile.jsx:3399-3412`, `staffordersmobile.jsx:3427-3451`). FIX: recompute the shift cutoff from current time on a timer/state tick, or derive it inline from a live "now" value instead of memoizing it only on working-hours changes.

- [P2] Double "Стол" prefix regression is back in multiple live paths - the project docs already note that DB table names already contain the prefix (`BUGS.md:112-116`), but the current file still prepends `"Стол "` in `RequestCard` (`staffordersmobile.jsx:1485`), `OrderCard` (`staffordersmobile.jsx:1590`), and `MyTablesModal` (`staffordersmobile.jsx:2332`).
