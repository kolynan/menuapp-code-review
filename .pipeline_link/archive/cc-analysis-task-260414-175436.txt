### Q4: Delay value (120ms) — sufficient?

**120ms is likely sufficient but borderline.** Analysis:

- B44 rate limit triggers at 4 concurrent calls (per S270 diagnostic).
- With sequential execution + 120ms gap, there's only ever 1 in-flight request.
- However, if `onSettled` fires invalidateQueries mid-batch (before skipInvalidate is implemented), the invalidation refetch could count against the rate limit too.
- With skipInvalidate, 120ms should be safe even for 30-item batches (30 * 120ms = 3.6s total).

**p-limit concurrency=2 alternative:** Would cut batch time roughly in half (~1.8s for 30 items) but adds complexity and still risks 429 if B44 counts refetch queries too. NOT recommended over sequential for V1.

**Recommendation: Keep 120ms for V1.** If testing shows it works for 30+ items, great. If not, bump to 150-200ms. The delay is a named constant in `runBatchSequential` defaults, so easy to tune.

### Q5: Sequential vs Promise.all chunks

Sequential `for-of` with `mutateAsync` is the correct architectural choice:
- Simplest to understand and debug
- Guaranteed no concurrency
- 4 items * 120ms delay = 480ms + ~4 * 200ms network = ~1.3s total — acceptable for a batch button
- 30 items = ~9.6s — getting long but still acceptable with proper visual feedback
- Promise.all chunks would save time but add complexity for marginal gain

**Verdict: Sequential is correct for V1.** Consider chunks only if users complain about 10+ item batch speed.

### Q6: Invalidate strategy

The draft proposes `skipInvalidate` flag in mutation vars + single invalidate after batch. This is **sound and recommended**.

**Current problem confirmed:** `onSettled` at line 1894-1897 fires `invalidateQueries(["orders"])` AND `invalidateQueries(["servedOrders", group.id])` for EVERY item in batch. For 10 items = 20 invalidation calls during the batch — very likely to amplify 429.

**The "200ms flat" alternative** (from the open question) is WORSE because:
1. It doubles batch time (10 items: 2s -> 4s)
2. Each invalidation refetch still races with the next mutation
3. It doesn't solve the root cause — just increases gap

**Recommendation: `skipInvalidate` + single invalidate.** Also need to handle `updateRequestMutation.onSuccess` (line 3515) the same way — it calls `invalidateQueries(["serviceRequests"])` per item.

**Important detail the draft missed:** `advanceMutation.onMutate` (line 1884-1889) calls `queryClient.cancelQueries({ queryKey: ["orders"] })` on EVERY mutation call. During a batch, this means item N's `onMutate` cancels the query that item N-1's onSettled just tried to refetch. With sequential execution this is actually beneficial (prevents mid-batch refetch), but it should be documented as intentional behavior.

### Q7: Undo window timing

**PARTIALLY CORRECT.** In the draft's Change A AFTER code:

```javascript
await runBatchSequential(actionable, async ({ order, payload }) => {
  ...
});
if (actionable.every(({ meta }) => meta.willServe)) {
  startUndoWindow(actionable.map(({ order }) => order), rowId);
}
```

`startUndoWindow` IS called AFTER the batch completes (after `await runBatchSequential`). This is correct.

**BUT:** The undo window starts a 3-second timeout (line 1925). If the batch took 3+ seconds (30 items), the undo window starts late but with full 3s — that's fine.

**HOWEVER:** There's a subtle issue — `startUndoWindow` at line 1920 checks `orders.length === 0` and returns early. The orders passed are the original `actionable` orders, which should be non-empty. OK.

**Real concern:** During `runBatchSequential`, each item is already committed to B44 backend via `mutateAsync`. The optimistic update in `onMutate` already moved them in the UI. If the user clicks "Undo" after batch, the onUndo handler (Change B) runs `runBatchSequential` to revert — but during the ORIGINAL batch, the UI already shows items as served (optimistic). If batch takes 3s, user sees "served" for 3s before undo toast appears. This is a UX concern but not a bug.

**Rating: OK** — timing is correct in the code flow.

### Q8: Request fan-out (onCloseRequest prop)

**The draft correctly identifies the problem:** `onCloseRequest` is a prop that wraps `updateRequestMutation.mutate(...)` (line 4399). Simple `forEach -> onCloseRequest(id)` fans out.

**Two options analyzed:**

Option 1 (draft's choice): Change `onCloseRequest` prop to return `mutateAsync` promise, use `await` inside `runBatchSequential`.
- Pros: Minimal changes, works with existing architecture
- Cons: Changes prop contract — all callers must handle the returned Promise (or ignore it)

Option 2: New `onBatchRequestAction(ids, status, extraFields)` prop with serialization in parent.
- Pros: Cleaner separation, single-item callers unchanged
- Cons: More code, new prop, parent component changes

**Recommendation: Option 1 (draft's choice) is correct.** The `.mutate()` -> `.mutateAsync()` change is safe because:
- `.mutateAsync()` returns a Promise that resolves/rejects
- Existing single-item callers at line 658 and 1273 call `onCloseRequest(id, status, ...)` and don't use the return value — returning a Promise instead of undefined is harmless (ignored)
- Line 4324 `onAction` is a different prop for single requests — NOT affected

**One concern:** The `isRequestPending` prop (line 4404) is `updateRequestMutation.isPending`. This becomes `true` on first call in the batch, so subsequent calls in `runBatchSequential` would see `isPending = true` — but since we're using `mutateAsync` in a sequential loop (not relying on React re-renders between calls), this is fine. The `isRequestPending` check is only in the `disabled` attribute of buttons, which won't re-render during the async loop.

### Q9: Scope creep risks

1. **`handleSingleAction` (line 1966):** Calls `handleOrdersAction([order], rowId)` with a single-item array. With the batch changes, this now: (a) sets `batchInFlight = true`, (b) calls `runBatchSequential` with 1 item (no delay since `i < items.length - 1` is false), (c) awaits mutateAsync, (d) sets `batchInFlight = false`. **Risk: adds ~1 render cycle overhead vs direct mutate.** The `await mutateAsync` adds a Promise resolution tick. **Impact: negligible** — single mutateAsync resolves at same speed as mutate. **No regression.**

2. **Single-request onClick (line 658, 1273):** These call `onCloseRequest(id, status, ...)` directly (not in forEach). With mutateAsync change, they return a Promise that nobody awaits. **No regression.**

3. **`isRateLimitError` (line 516-520):** Draft says "DO NOT remove" — correct. The `runBatchSequential` helper uses it for early `break`. The existing `updateRequestMutation.onError` (line 3516-3521) still fires for individual 429s. No conflict.

4. **Guest batch loading (line 3603):** `Promise.all(guestPromises)` for `SessionGuest.get()` reads — NOT mutations, different entity, lower priority. Could hit read rate limits on large tables but out of scope for this bug.

### Q10: Regressions to verify

The draft's regression checklist is **good but incomplete**. Missing items:

- **(g) Legacy block batch buttons (lines 759, 760, 1374, 1375, 2349-2351):** These call `handleOrdersAction` too. The fix to `handleOrdersAction` covers them automatically, but they should be explicitly tested since they're in a different render path (non-hall / compact-table modes).

- **(h) handleCloseAllOrders (line 4132):** Since this is NOT covered by the fix, it remains a 429 risk when closing tables with many orders. Should be noted as known-gap.
