# Code Review Report: `pages/Profile/base/profile.jsx`

## Summary

The base Profile file has several regressions against the documented Profile behavior and the current PartnerShell contract. The most important problems are incorrect role/restaurant resolution for invited partner staff, role-label i18n failures, and lost mobile-first fixes for the save action and input touch targets.

### Critical Issues (P0) â€” Must Fix

None.

### High Priority (P1) â€” Should Fix

- [P1] Profile bypasses `PartnerShell` context and can show the wrong role/restaurant - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L73) re-fetches `base44.auth.me()` and then resolves the restaurant from `userData.partner`, even though the page is always rendered inside [`PartnerShell`](C:/Dev/menuapp-code-review/pages/PartnerShell/260301-00%20PartnerShell%20RELEASE.jsx#L362), which already computes the effective `currentUser`, `userRole`, `partnerId`, and `partner`. For invited `director` / `managing_director` users whose `User.partner` is empty, this page can incorrectly show "not attached" and a mismatched role even while the shell has a valid partner context. FIX: Read `currentUser`, `userRole`, and `partner` from `usePartnerAccess()` inside `ProfileContent`, keep only the editable `fullName` state local, and remove the duplicate `auth.me()` / `Partner.get()` load path.

- [P1] Role badge can be blank or leak raw DB enum values - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L138) returns `tr(\`profile.role.${userRole}\`, userRole)`. If `userRole` is missing, the fallback is `undefined`, so the badge renders with no label. If the translation key is missing, the fallback exposes raw internal values such as `partner_owner` to users. This is a direct i18n/UI regression. FIX: Add an explicit null guard returning `tr("profile.role.unknown", "...")`, and remove the raw-enum fallback so unresolved roles do not surface DB values.

### Medium Priority (P2) â€” Recommended

- [P2] Hardcoded Russian fallbacks break non-Russian fallback behavior - The local `tr()` helper in [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L41) depends on hardcoded Russian literals passed at every call site, for example [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L126). If an English or Kazakh translation key is missing, the page silently mixes Russian copy into the UI instead of using catalog-driven translations. FIX: Remove language-specific literals from component code and rely on locale keys; if a safety fallback is still needed, use a locale-neutral helper like the current `pages/Profile/profile.jsx` version instead of Russian inline strings.

# Code Review Report: `pages/Profile/base/profile.jsx`

## Summary

The base Profile file still contains the same regressions already hinted at in the page notes: it duplicates PartnerShell data loading, has multiple i18n contract violations, and has lost several mobile-first fixes that exist in the later Profile variants. No P0 crash was found in this file, but there are several P1/P2 issues that affect correctness and user-facing behavior.

## Findings

- [P1] Profile bypasses PartnerShell context and can show the wrong role or restaurant - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L73) reloads `base44.auth.me()` and then resolves the restaurant from `userData.partner` instead of using the effective partner context already provided by [`PartnerShell`](C:/Dev/menuapp-code-review/pages/PartnerShell/260301-00%20PartnerShell%20RELEASE.jsx). For cabinet-access users such as `director` and `managing_director`, `User.partner` can be empty while the shell already resolved a valid partner, so this page can incorrectly show "not attached" and the wrong role. FIX: consume `usePartnerAccess()` inside `ProfileContent` for `currentUser`, `userRole`, `partnerId`, and `partner`; keep only editable `fullName` state and the `updateMe()` mutation local.

- [P1] Hardcoded fallback strings violate the i18n contract and mix languages - The page passes inline Russian and English literals into `tr()` throughout the file, for example at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L126), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L165), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L243). The repo rules require user-facing copy to come from `t('key')`, and these hardcoded fallbacks can silently switch the UI into the wrong language when a key is missing. FIX: remove inline user-facing fallback literals from component code and either rely on locale keys only or use a locale-neutral missing-key helper.

- [P1] Role badge can be blank or expose raw internal enum values - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L138) returns `tr(\`profile.role.${userRole}\`, userRole)`. If `userRole` is absent, the badge label becomes `undefined` and renders blank. If the translation key is missing, raw values such as `partner_owner` leak to the UI. FIX: add an explicit null guard that returns `tr("profile.role.unknown", "...")`, and never use the raw enum value as the fallback label.

- [P2] Partner load failure is indistinguishable from "no restaurant" - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L93) swallows `Partner.get()` errors, and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L206) always falls back to `profile.no_restaurant`. That hides a real backend failure behind a misleading state. FIX: add `isPartnerLoadFailed` state, set it in the partner-fetch `catch`, and render `profile.restaurant_load_error` when the lookup fails.

- [P2] Primary save action regressed from fixed mobile footer to inline card button - The save button is back inside the form at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L277). On small screens it can fall below the fold again, undoing the page's documented mobile fix and making the primary action easy to miss. FIX: restore the `fixed inset-x-0 bottom-0` footer with safe-area padding and reserve space in the content area for it.

- [P2] Full Name and Email inputs lost the 44px touch-target fix - The inputs at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L234) and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L244) no longer have the documented `min-h-[44px]` mobile sizing. That regresses tap usability on phones. FIX: add `className="min-h-[44px]"` to both inputs while preserving the email field's read-only styling.

- [P3] PartnerShell wrapper omits `activeTab="profile"` - The wrapper at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L303) renders plain `<PartnerShell>` even though the other Profile variants in this folder use `activeTab="profile"`. That can leave the shell nav without a correct active state. FIX: render `<PartnerShell activeTab="profile">`.

- [P3] Nested `min-h-screen` containers create extra viewport height inside PartnerShell - The loading, error, and ready roots at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L152), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L163), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L209) each request a full viewport height even though PartnerShell already provides a full-height layout. On mobile this adds unnecessary scroll space. FIX: replace inner `min-h-screen` usage with `h-full`, `w-full`, or a smaller local min-height that fits the shell content area.

- [P3] Load error state is not announced to assistive tech - The error container at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L163) replaces the loading UI but has no `role="alert"`. Screen-reader users may not be notified that loading ended with an error. FIX: add `role="alert"` to the error-state container.

- [P3] Decorative icons are exposed to screen readers - Icons such as [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L164), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L171), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L186), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L194), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L218) are decorative but lack `aria-hidden="true"`. +# Code Review Report: `pages/Profile/base/profile.jsx`
+
+## Summary
+
+The base Profile file still contains the same regressions already hinted at in the page notes: it duplicates PartnerShell data loading, has multiple i18n contract violations, and has lost several mobile-first fixes that exist in the later Profile variants. No P0 crash was found in this file, but there are several P1/P2 issues that affect correctness and user-facing behavior.
+
+## Findings
+
+- [P1] Profile bypasses PartnerShell context and can show the wrong role or restaurant - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L73) reloads `base44.auth.me()` and then resolves the restaurant from `userData.partner` instead of using the effective partner context already provided by [`PartnerShell`](C:/Dev/menuapp-code-review/pages/PartnerShell/260301-00%20PartnerShell%20RELEASE.jsx). For cabinet-access users such as `director` and `managing_director`, `User.partner` can be empty while the shell already resolved a valid partner, so this page can incorrectly show "not attached" and the wrong role. FIX: consume `usePartnerAccess()` inside `ProfileContent` for `currentUser`, `userRole`, `partnerId`, and `partner`; keep only editable `fullName` state and the `updateMe()` mutation local.
+
+- [P1] Hardcoded fallback strings violate the i18n contract and mix languages - The page passes inline Russian and English literals into `tr()` throughout the file, for example at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L126), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L165), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L243). The repo rules require user-facing copy to come from `t('key')`, and these hardcoded fallbacks can silently switch the UI into the wrong language when a key is missing. FIX: remove inline user-facing fallback literals from component code and either rely on locale keys only or use a locale-neutral missing-key helper.
+
+- [P1] Role badge can be blank or expose raw internal enum values - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L138) returns `tr(\`profile.role.${userRole}\`, userRole)`. If `userRole` is absent, the badge label becomes `undefined` and renders blank. If the translation key is missing, raw values such as `partner_owner` leak to the UI. FIX: add an explicit null guard that returns `tr("profile.role.unknown", "...")`, and never use the raw enum value as the fallback label.
+
+- [P2] Partner load failure is indistinguishable from "no restaurant" - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L93) swallows `Partner.get()` errors, and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L206) always falls back to `profile.no_restaurant`. That hides a real backend failure behind a misleading state. FIX: add `isPartnerLoadFailed` state, set it in the partner-fetch `catch`, and render `profile.restaurant_load_error` when the lookup fails.
+
+- [P2] Primary save action regressed from fixed mobile footer to inline card button - The save button is back inside the form at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L277). On small screens it can fall below the fold again, undoing the page's documented mobile fix and making the primary action easy to miss. FIX: restore the `fixed inset-x-0 bottom-0` footer with safe-area padding and reserve space in the content area for it.
+
+- [P2] Full Name and Email inputs lost the 44px touch-target fix - The inputs at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L234) and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L244) no longer have the documented `min-h-[44px]` mobile sizing. That regresses tap usability on phones. FIX: add `className="min-h-[44px]"` to both inputs while preserving the email field's read-only styling.
+
+- [P3] PartnerShell wrapper omits `activeTab="profile"` - The wrapper at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L303) renders plain `<PartnerShell>` even though the other Profile variants in this folder use `activeTab="profile"`. That can leave the shell nav without a correct active state. FIX: render `<PartnerShell activeTab="profile">`.
+
+- [P3] Nested `min-h-screen` containers create extra viewport height inside PartnerShell - The loading, error, and ready roots at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L152), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L163), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L209) each request a full viewport height even though PartnerShell already provides a full-height layout. On mobile this adds unnecessary scroll space. FIX: replace inner `min-h-screen` usage with `h-full`, `w-full`, or a smaller local min-height that fits the shell content area.
+
+- [P3] Load error state is not announced to assistive tech - The error container at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L163) replaces the loading UI but has no `role="alert"`. Screen-reader users may not be notified that loading ended with an error. FIX: add `role="alert"` to the error-state container.
+
+- [P3] Decorative icons are exposed to screen readers - Icons such as [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L164), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L171), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L186), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L194), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L218) are decorative but lack `aria-hidden="true"`. FIX: mark decorative icons as `aria-hidden="true"` and rely on nearby text for the accessible name.

+# Code Review Report: `pages/Profile/base/profile.jsx`
+
+## Summary
+
+The base Profile file still contains the same regressions already hinted at in the page notes: it duplicates PartnerShell data loading, has multiple i18n contract violations, and has lost several mobile-first fixes that exist in the later Profile variants. No P0 crash was found in this file, but there are several P1/P2 issues that affect correctness and user-facing behavior.
+
+## Findings
+
+- [P1] Profile bypasses PartnerShell context and can show the wrong role or restaurant - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L73) reloads `base44.auth.me()` and then resolves the restaurant from `userData.partner` instead of using the effective partner context already provided by [`PartnerShell`](C:/Dev/menuapp-code-review/pages/PartnerShell/260301-00%20PartnerShell%20RELEASE.jsx). For cabinet-access users such as `director` and `managing_director`, `User.partner` can be empty while the shell already resolved a valid partner, so this page can incorrectly show "not attached" and the wrong role. FIX: consume `usePartnerAccess()` inside `ProfileContent` for `currentUser`, `userRole`, `partnerId`, and `partner`; keep only editable `fullName` state and the `updateMe()` mutation local.
+
+- [P1] Hardcoded fallback strings violate the i18n contract and mix languages - The page passes inline Russian and English literals into `tr()` throughout the file, for example at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L126), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L165), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L243). The repo rules require user-facing copy to come from `t('key')`, and these hardcoded fallbacks can silently switch the UI into the wrong language when a key is missing. FIX: remove inline user-facing fallback literals from component code and either rely on locale keys only or use a locale-neutral missing-key helper.
+
+- [P1] Role badge can be blank or expose raw internal enum values - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L138) returns `tr(\`profile.role.${userRole}\`, userRole)`. If `userRole` is absent, the badge label becomes `undefined` and renders blank. If the translation key is missing, raw values such as `partner_owner` leak to the UI. FIX: add an explicit null guard that returns `tr("profile.role.unknown", "...")`, and never use the raw enum value as the fallback label.
+
+- [P2] Partner load failure is indistinguishable from "no restaurant" - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L93) swallows `Partner.get()` errors, and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L206) always falls back to `profile.no_restaurant`. That hides a real backend failure behind a misleading state. FIX: add `isPartnerLoadFailed` state, set it in the partner-fetch `catch`, and render `profile.restaurant_load_error` when the lookup fails.
+
+- [P2] Primary save action regressed from fixed mobile footer to inline card button - The save button is back inside the form at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L277). On small screens it can fall below the fold again, undoing the page's documented mobile fix and making the primary action easy to miss. FIX: restore the `fixed inset-x-0 bottom-0` footer with safe-area padding and reserve space in the content area for it.
+
+- [P2] Full Name and Email inputs lost the 44px touch-target fix - The inputs at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L234) and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L244) no longer have the documented `min-h-[44px]` mobile sizing. That regresses tap usability on phones. FIX: add `className="min-h-[44px]"` to both inputs while preserving the email field's read-only styling.
+
+- [P3] PartnerShell wrapper omits `activeTab="profile"` - The wrapper at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L303) renders plain `<PartnerShell>` even though the other Profile variants in this folder use `activeTab="profile"`. That can leave the shell nav without a correct active state. FIX: render `<PartnerShell activeTab="profile">`.
+
+- [P3] Nested `min-h-screen` containers create extra viewport height inside PartnerShell - The loading, error, and ready roots at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L152), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L163), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L209) each request a full viewport height even though PartnerShell already provides a full-height layout. On mobile this adds unnecessary scroll space. FIX: replace inner `min-h-screen` usage with `h-full`, `w-full`, or a smaller local min-height that fits the shell content area.
+
+- [P3] Load error state is not announced to assistive tech - The error container at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L163) replaces the loading UI but has no `role="alert"`. Screen-reader users may not be notified that loading ended with an error. FIX: add `role="alert"` to the error-state container.
+
+- [P3] Decorative icons are exposed to screen readers - Icons such as [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L164), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L171), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L186), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L194), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L218) are decorative but lack `aria-hidden="true"`. FIX: mark decorative icons as `aria-hidden="true"` and rely on nearby text for the accessible name.
