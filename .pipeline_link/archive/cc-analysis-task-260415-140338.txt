### 1.3 Root Cause Analysis ‚Äî REVISED

**The original hypothesis was WRONG.** The Completed tab does NOT filter by `Table.status`. It filters by order state within each group. A table moves to Completed when ALL its orders are either:
- `status === 'closed'` (closed by closeSession)
- `status === 'cancelled'`
- At `isFinishStage` AND `status === 'closed'` or `'cancelled'`

**So after `closeSession()` runs:**
1. `closeSession` bulk-closes all non-cancelled orders to `status: 'closed'` (line 169)
2. The tab filter should then see: `hasActiveOrder = false`, `hasServedButNotClosed = false`
3. The table SHOULD move to Completed ‚Äî **unless there are active requests**

**KEY FINDING:** The bug is likely caused by `hasActiveRequest`:
```js
const hasActiveRequest = group.type === 'table' && activeRequests.some(r => getLinkId(r.table) === group.id);
```
`activeRequests` is filtered at lines 3539‚Äì3546:
```js
return !["done", "cancelled"].includes(r.status);
```
If there's an **open ServiceRequest** still linked to that table (e.g., status "new" or "accepted" or "in_progress"), the table stays in Active tab even after closeSession. `closeSession()` does NOT close ServiceRequests ‚Äî only Orders.

**Alternative scenario:** There could be a **query cache staleness** issue. After `closeSession`, the code calls `refetchOrders()` (line 4151), but:
- It does NOT invalidate `serviceRequests` query
- It does NOT invalidate `tables` query
- React-query cache might still show old order states briefly

**Possible scenario 3:** The `closeSession` uses `Promise.all` (line 166) which fires all Order updates in parallel ‚Äî this could trigger 429 rate limits on B44, causing some orders to fail silently. If even one order doesn't get `status: 'closed'`, the table stays Active.

### 1.4 Fix Recommendations

**Option A: Add `Table.status` update in `closeSession()`**
- NOT recommended yet. The tab filtering does NOT use `Table.status` at all. Adding it would be forward-looking (for future features) but would NOT fix the current bug.
- Also, `closeSession` only has `sessionId`, not `tableId`. Would need: `const session = await base44.entities.TableSession.filter({ id: sessionId }); const tableId = getLinkId(session[0]?.table);` + `base44.entities.Table.update(tableId, { status: 'closed' })`.

**Option B: Fix the actual bug ‚Äî close ServiceRequests in `closeSession()`**
- RECOMMENDED if root cause is open requests. Add to `closeSession()`:
  ```js
  const tableSession = await base44.entities.TableSession.filter({ id: sessionId });
  const tableId = getLinkId(tableSession[0]?.table);
  if (tableId) {
    const requests = await base44.entities.ServiceRequest.filter({ table: tableId });
    await Promise.all(
      requests.filter(r => !['done', 'cancelled'].includes(r.status))
        .map(r => base44.entities.ServiceRequest.update(r.id, { status: 'done' }))
    );
  }
  ```

**Option C: Fix at the call-site (line 4148) ‚Äî also invalidate serviceRequests**
- SAFE quick fix. After `closeSession()`, add `refetchRequests()`:
  ```js
  // At line 4151, add:
  if (!isKitchen) refetchRequests();
  ```
  Wait ‚Äî this already exists at line 4131 in `handleRefresh`, but NOT in `confirmCloseTable`. Adding it ensures the request cache is refreshed.

**Option D: Use `runBatchSequential` instead of `Promise.all` in `closeSession()`**
- Fixes the 429 risk in bulk close. But `runBatchSequential` is defined only in the RELEASE file, not in sessionHelpers.js.

**RECOMMENDED approach:** Two-part fix:
1. **Fix C** (quick): Add `refetchRequests()` after `closeSession()` in `confirmCloseTable` (line 4151) to clear stale request cache.
2. **Fix B** (thorough): Add ServiceRequest closure in `closeSession()` so requests don't keep the table in Active.
3. **Option A** (bonus): Add `Table.status = 'closed'` for forward-compat (B44_Entity_Schemas v3.0 has this field).

### 1.5 Regression Risks

1. **Other callers of `closeSession()`**: Only one call-site (line 4148). No regression risk from other callers.
2. **Rate limit from bulk operations**: `closeSession` uses `Promise.all` ‚Äî if a table has many orders + requests, this fires many parallel mutations and could trigger 429. Should switch to sequential pattern.
3. **UI relying on old Table.status semantics**: Currently no code reads `Table.status` in SOM. But PartnerTables page might. Need to check before adding `Table.status = 'closed'`.
4. **ServiceRequest closure side effects**: If we auto-close requests, the waiter won't see "close request" button animations or get notified. Probably fine since the table is closing anyway.
5. **Query invalidation timing**: `refetchOrders()` at line 4151 triggers before React processes the close ‚Äî this is correct (re-fetches fresh data showing closed orders).

### 1.6 Test Plan

1. **Waiter opens SOM** ‚Üí sees table with 1+ orders in "Active" tab ‚Üí clicks all orders through to served ‚Üí clicks "Close Table" ‚Üí confirms ‚Üí table should disappear from Active and appear in "Completed" tab.
2. **Same flow with ServiceRequests**: Waiter opens table with 2 orders AND a service request ‚Üí advances orders ‚Üí closes table ‚Üí table should still move to Completed (request should auto-close or at least not block the move).
3. **Edge case ‚Äî partial close**: Table with 5 orders, 2 cancelled ‚Üí Close Table ‚Üí verify all 3 non-cancelled orders become `status: 'closed'` AND table moves to Completed.

### 2.3 Analysis of `onSuccess` / `onSettled` invalidation patterns

**updateStatusMutation (line 1594‚Äì1607):**
```js
onSettled: () => queryClient.invalidateQueries({ queryKey: ["orders"] }),
```
- ALWAYS fires ‚Äî no `__batch` check.
- Every single tap on an OrderCard action button = 1 mutation + 1 full orders refetch.
- If a waiter taps 3 OrderCards quickly = 3 mutations + 3 refetches simultaneously.

**advanceMutation (line 1904‚Äì1921):**
```js
onSettled: (_data, _err, vars) => {
  if (vars?.__batch) return;
  queryClient.invalidateQueries({ queryKey: ["orders"] });
  queryClient.invalidateQueries({ queryKey: ["servedOrders", group.id] });
},
```
- Has `__batch` guard ‚Äî batch calls skip invalidation.
- But note: `handleSingleAction` at line 2002 calls `handleOrdersAction([order], rowId)` which uses `__batch: true` (line 1990). So single actions through the TABLE CARD path DO use `__batch` and then invalidate once after the batch at line 1992-1993. This is correct.

**The issue is that `updateStatusMutation` (OrderCard's own mutation at line 1594) does NOT have `__batch` support ‚Äî it fires `onSettled` unconditionally.**

There are TWO DIFFERENT mutation objects:
1. `updateStatusMutation` in `OrderCard` component (line 1594) ‚Äî simple, no `__batch`. Used in kitchen view for individual order cards.
2. `advanceMutation` in `TableOrderGroup` component (line 1904) ‚Äî has `__batch` guard. Used in hall view grouped cards.

### 2.4 Rate Limit Vectors ‚Äî Ranked by Severity

**HIGH:**
1. **[D] Batch request close via `forEach` (lines 657, 1272)** ‚Äî fires N mutations synchronously without `runBatchSequential`. Each mutation triggers `invalidateQueries`. If table has 4 requests: 4 mutations + 4 invalidations in <50ms. VERY likely to hit 429.
2. **[F] `closeSession` `Promise.all` (sessionHelpers.js line 166)** ‚Äî fires all order closes in parallel. 5 orders = 5 simultaneous B44 API calls. No throttling at all.

**MEDIUM:**
3. **[A] `updateStatusMutation.onSettled` (line 1606)** ‚Äî every single-order action in OrderCard fires full invalidation. Low risk for single taps, but if waiter is fast or kitchen view has many items, quick taps can stack.

**LOW:**
4. **[B,C] Single request actions (lines 4360, 4435)** ‚Äî one mutation per click, manageable.

### 2.5 Proposed Minimal Fix Pattern

**Fix for [D] ‚Äî Replace `forEach` with `runBatchSequential` + `__batch`:**
```js
// BEFORE (line 657):
onClick={() => tableRequests.forEach(r => onCloseRequest(r.id, 'accepted', {...}))}

// AFTER:
onClick={async () => {
  await runBatchSequential(tableRequests, (r) =>
    onBatchCloseRequestAsync(r.id, 'accepted', { assignee: effectiveUserId, assigned_at: new Date().toISOString() })
  );
  queryClient.invalidateQueries({ queryKey: ["serviceRequests"] });
}}
```
**Note:** The expanded card view at line 2369 ALREADY uses this correct pattern with `runBatchSequential` + `onBatchCloseRequestAsync`. The collapsed card views at lines 657 and 1272 use the old `forEach` pattern ‚Äî these were not updated during the S272 rate limit fix.

**Fix for [F] ‚Äî Replace `Promise.all` in `closeSession` with sequential pattern:**
Either:
(a) Import/inline `runBatchSequential` in sessionHelpers.js
(b) Use a simple for-loop with delay:
```js
for (const o of ordersToClose) {
  await base44.entities.Order.update(o.id, { status: 'closed' });
  if (ordersToClose.indexOf(o) < ordersToClose.length - 1) {
    await new Promise(r => setTimeout(r, 120));
  }
}
```

**Fix for [A] ‚Äî Add `__batch` guard to `updateStatusMutation`:**
Not strictly needed since OrderCard only fires single mutations, but for safety:
```js
onSettled: (_data, _err, vars) => {
  if (vars?.__batch) return;
  queryClient.invalidateQueries({ queryKey: ["orders"] });
},
```

## Section 4 ‚Äî Risks Outside Scope (max 5)

1. **[P1] `closeSession` `Promise.all` (sessionHelpers.js:166)** ‚Äî bulk Order.update in parallel, no throttling, no error handling per-item. If one order fails, `Promise.all` rejects and remaining orders stay open. Should use `Promise.allSettled` or sequential.

2. **[P2] Hardcoded Russian strings** ‚Äî Lines 4149 ("–°—Ç–æ–ª –∑–∞–∫—Ä—ã—Ç"), 4153 ("–û—à–∏–±–∫–∞ –ø—Ä–∏ –∑–∞–∫—Ä—ã—Ç–∏–∏"), 4163 ("–ù–µ—Ç —ç—Ç–∞–ø–∞ '–ó–∞–≤–µ—Ä—à—ë–Ω'"), 4378 ("–ù–µ—Ç –∑–∞–≤–µ—Ä—à—ë–Ω–Ω—ã—Ö –∑–∞–∫–∞–∑–æ–≤"), etc. ~15+ user-facing strings not using `t()` / i18n.

3. **[P2] `updateStatusMutation` vs `advanceMutation` duplication** ‚Äî Two separate Order.update mutations with different onMutate/onSettled behaviors. If one gets patched (e.g., for rate limit), the other doesn't. Should consolidate or at least align patterns.

4. **[P1] `closeSession` does NOT invalidate any query caches** ‚Äî After closing session + orders + (future) requests, the SOM UI relies on `refetchOrders()` at the call-site (line 4151). But if `closeSession` is called from elsewhere in the future, the caller must remember to refetch. The function should return metadata (which orders were closed) or invalidate caches itself.

5. **[P2] `confirmCloseTable` missing error details** ‚Äî Line 4153 shows generic "–û—à–∏–±–∫–∞ –ø—Ä–∏ –∑–∞–∫—Ä—ã—Ç–∏–∏" without logging `err`. If `closeSession` fails due to 429 (from `Promise.all`), the waiter has no idea what happened. Should log error and ideally retry.
