## CRITICAL FINDINGS (would cause wrong implementation)

### C1. Fix 1 — `getSafeStatus()` not mentioned (P0 — missed function)

The prompt says to update or remove `getOrderStatus()`. But:
- `getOrderStatus` is a **PROP** (line 53), defined outside CartView.jsx. Cannot be modified here.
- The actual status-label function IN CartView is `getSafeStatus()` (lines 275-321).
  It has a full 5-status fallback map with 'Отправлено', 'Готовится', 'Готов', 'Принят', 'Подано'.
- `getSafeStatus` is called at line 788: `const status = getSafeStatus(getOrderStatus(order))`
- Prompt never mentions `getSafeStatus` at all.

**Fix for prompt:** Add: "Grep: `getSafeStatus` — this is the IN-FILE function that maps status
codes to display labels. Either reduce to 2 labels or remove if bucket rendering no longer needs
per-order status. `getOrderStatus` is a PROP — do NOT modify it, but its call sites may change."

### C2. Fix 1 — `Новый заказ` rename to `В корзине` not in grep targets

Prompt says CV-49 renames "Новый заказ" to "В корзине". But the grep targets in Fix 1's
"Что менять" section don't include `Новый заказ`. The actual text is at:
- Line 1011: `{tr('cart.new_order', 'Новый заказ')}`

**Fix for prompt:** Add grep target: `Grep: 'Новый заказ' -> rename to 'В корзине'` and
update the tr() key from `cart.new_order` to `cart.group.in_cart`.

### C3. Fix 2 — Wrong grep patterns for price variables

Prompt says: `Grep: totalPrice|grandTotal|orderTotal|cartTotal`
Actual variable names in code:

### C4. Fix 2 — `Сумма заказа` and `К отправке` don't exist in code

### C5. Fix 3 — `submitPhase` state machine DUPLICATES existing props

The code already has a 3-state CTA:
- `isSubmitting` prop → "Отправляем..." (line 1142-1143)
- `submitError` prop → "Повторить отправку" (line 1144-1145)
- Default → "Отправить заказ официанту" (line 1146)

Plus dedicated error display block (lines 1060-1067).

The prompt's `submitPhase` useState('idle'|'submitting'|'success'|'error') would create
DUPLICATE state for submitting and error, conflicting with existing props.

**Fix for prompt:** Clarify: "Only the SUCCESS feedback phase is missing. Do NOT duplicate
`isSubmitting`/`submitError` props. Add `submitPhase` with values `'idle'|'success'` only.
Use existing `isSubmitting` prop for submitting state, existing `submitError` prop for error.
The new state only tracks the 1.5s success window between isSubmitting->false and idle."

### C6. Fix 7 — Header merge ALREADY DONE

Lines 680-723 already show table + guest on line 1 (`{tableLabel} . {guestDisplay}`)
and order count + sum on line 2. CV-31 was already implemented.

The only actual change needed: line 714 counts ORDERS (`todayMyOrders.length`)

**Fix for prompt:** Replace Fix 7's "merge table + guest onto line1" (already done) with:

to 'блюдо/блюда/блюд'."

## MODERATE FINDINGS (would cause confusion or suboptimal code)

### M1. Fix 1 — `bucketDisplayNames` has hardcoded Russian (i18n violation)

**Fix:** Add: "Replace bucketDisplayNames with tr() calls: `served: tr('cart.group.served', 'Выдано')`,
`in_progress: tr('cart.group.in_progress', 'В работе')`."

### M2. Fix 1 — Cancelled orders already filtered (prompt redundant warning)

Line 393: `filter(o => o.status !== 'cancelled')` — cancelled orders are filtered in
`todayMyOrders` BEFORE bucket creation. Prompt's bucket code says `if (s !== 'cancelled')`
as a safety check — this is redundant but harmless. Worth noting to avoid confusion.

### M3. Fix 1 — Empty drawer state already handled

Lines 848-863: The `isV8` state already shows "Ничего не ждёте" when only served orders exist.
Prompt asks to "verify that the drawer cannot be opened when empty OR show empty state" —
the empty-all case (0 orders, 0 cart) is actually handled by StickyCartBar not showing the
drawer trigger. But there's no explicit empty state IN the drawer. Worth adding a simple
message for defensive robustness.

### M4. Fix 3 — D12 hook reuse guidance is misleading

Prompt says "prefer REUSING dead hook slots for submitPhase". But the 5-bucket
`expandedStatuses` is a SINGLE useState object (line 93), not 5 separate useStates.
Simplifying it to 2 keys doesn't free any hook slots. There are no dead hooks to reuse.

**Fix:** Remove D12 hook reuse guidance for this specific case, or clarify: "No hook slots
are freed by bucket simplification — expandedStatuses is one object. Place new useState
after line 127 (after `nowTs` hook)."

### M5. Fix 4 — Current auto-collapse is too aggressive

Lines 102-113 collapse ALL statuses (served, ready, in_progress, accepted) when cart > 0.
Prompt's Fix 4 only discusses served collapse but doesn't mention that the current code
also collapses non-served buckets. After Fix 1 reduces to 2 groups, the implementer needs
to know that the EXISTING useEffect at lines 102-113 must be rewritten, not just tweaked.

### M6. Fix 5 — `earnedPoints` vs `motivationPoints` confusion

Prompt says to show `earnedPoints` in cart group. But:
- `earnedPoints` (prop, line 39) — earned from PLACED orders
- `motivationPoints` (line 1119, calculated inline) — projected for CART items

The correct value for "В корзине" bonus display is `motivationPoints`, not `earnedPoints`.

**Fix:** Specify: "Use the same calculation as line 1119:
`Math.round(cartTotalAmount * (partner?.loyalty_points_per_currency || 1))`."

### M7. Fix 6 — Cart qty display format mismatch

Prompt says cart should show "Суши x2  89.54". Actual code (line 1021) shows price breakdown:
`{formatPrice(item.price)} x {item.quantity}` (e.g., "44.77 x 2").
This is a different format from the prompt's wireframe. Clarify intended behavior.

### M8. Fix 2 — Header total verification incomplete

Prompt correctly says "Header total must include items from ALL three sources."
Current `ordersSum` (line 417-420) only sums todayMyOrders. Adding cart items requires
creating a new combined total: `ordersSum + (cartTotalAmount || 0)`. Prompt should
provide the exact formula.
