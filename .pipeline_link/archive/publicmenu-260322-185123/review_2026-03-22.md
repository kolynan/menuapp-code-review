## Code Review Report: PublicMenu (`x.jsx`, `CartView.jsx`)

### Summary
Current PublicMenu drawer flow still has broken bill-state wiring, split table-verification logic, and an error path that mislabels backend failures as bad QR links. Bottom-sheet mobile UX and i18n issues also remain in the current page code.

### Critical Issues (P0) — Must Fix
None.

### High Priority (P1) — Should Fix
- `CartView.jsx:99-225`, `x.jsx:3441-3479` — Table verification logic is split across two surfaces. `CartView` still owns attempt/cooldown state and a hidden auto-verify effect, while the visible input and CTA live in `x.jsx`. This can auto-submit verification unexpectedly, duplicate `verifyTableCode()` calls, and keep the lockout state invisible in the active bottom sheet.
- `x.jsx:3363-3367`, `CartView.jsx:52-82` — Bill/request-bill props are passed from `x.jsx` but never accepted or rendered by `CartView`. The drawer therefore cannot switch to post-send/open-bill layouts or expose the expected bill CTA from this surface.
- `x.jsx:1325-1345`, `x.jsx:3042-3046` — Partner lookup swallows backend errors and falls through to the same `partner_not_found` UI used for real missing partners. Transient API failures are shown as a bad QR link, with no retryable error state.

### Medium Priority (P2) — Recommended
- `CartView.jsx:441-495`, `x.jsx:3311-3315` — The cart drawer header still uses an `X` close button, the control has no `aria-label`, the tap target is smaller than 44x44, and the header scrolls away because it sits inside the main drawer scroll container instead of staying sticky.
- `x.jsx:3405-3414` — The table-confirmation bottom sheet header has no explicit close/minimize control. Dismissal depends on swipe/backdrop only, which is inconsistent with the cart drawer and weaker on mobile.
- `x.jsx:2389-2414` — `checkRateLimit()` fails open. Any exception in the order lookup returns `true`, which silently disables anti-spam protection and still lets the submit path continue.

### Low Priority (P3) — Nice to Have
- `CartView.jsx:530-535` — The reward-email input hardcodes `placeholder="email@example.com"` instead of using the same translation path already used later for the loyalty email field.
