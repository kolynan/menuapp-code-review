## Key Decisions

### Q1: Option B — Lightweight Snapshot
Store { tableSessionId, sessionGuestId, orderIds[], lastSeenTs, pendingSubmit, schemaVersion }.
Not full snapshot (stale data risk, reconcile complexity) nor minimal (too many round-trips).

### Q2: Parallel Fetch with 8s Global Timeout
Phase 1: validate session (single request).
Phase 2: parallel fetch guests + orders + items.
Transitions: S-loading -> S-restored / S-failed-network (8s timeout) / S-lost (404/410).

### Q3: Invalidation at 12h Client TTL
Server 404/410, expired session, 12h lastSeenTs TTL, schema version mismatch, table change.
12h chosen as balance between long dinners (4-5h) and next-day staleness.

### Q4: pendingSubmit Flag for Kill-During-Submit
Set before fetch, cleared after success/failure. On restore, compare timestamp with server orders.
False positive handling: informational banner, auto-dismiss.

### Q5: Polling After S-restored Only
No polling during S-loading or S-failed-network. Post-restore polling failure stays in S-restored
with stale data + toast (not regression to S-failed-network).
