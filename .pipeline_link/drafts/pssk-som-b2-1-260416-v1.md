# SOM Б2.1 — Fix BUG-SM-015: New Order on Closed Table

## Context
Файл: `pages/StaffOrdersMobile/staffordersmobile.jsx`
RELEASE: `pages/StaffOrdersMobile/260415-01 StaffOrdersMobile RELEASE.jsx` (4617 строк)
Задача: Новый заказ на закрытый стол должен создавать новую скс в «Активных», а не попадать в закрытую скс в «Завершённых»
Бюджет: $12 | Шаблон: consensus-with-discussion-v2 (С5v2)
BACKLOG: #BUG-SM-015 (P0), #347 (M)

## Root Cause (VERIFIED via code analysis)
Диагноз (⚠️ подтверждён чтением кода — НЕ предположения):

1. `openSessions` query (строка ~3541) имеет `staleTime: 30_000` — SOM не знает о новой TableSession до 30 секунд
2. `orderGroups` (строка ~3768) группирует hall-заказы по `tableId` — старые и новые заказы стола мёрджатся в одну карточку
3. `filteredGroups` определяет таб по `openSessionByTableId[group.id]` — если сессия не обнаружена, группа идёт в «Завершённые»

Когда гость заказывает на закрытый стол → PublicMenu создаёт новую TableSession через `getOrCreateSession()`. Но SOM видит её только после polling (~30 сек). Всё это время новые заказы попадают в старую карточку в «Завершённых».

## Fix A: Уменьшить staleTime для openSessions (30s → 5s)

### Проблема
SOM кэширует список открытых TableSession на 30 секунд. Когда гость создаёт новую сессию, официант видит правильный таб только через 30+ секунд. Всё это время новые заказы отображаются в «Завершённых».

### Что менять
Grep: `staleTime: 30_000`
Ожидаемое местонахождение: внутри `useQuery` для `openSessions` (строки ~3541-3552)

Текущий код:
```js
const { data: openSessions = [] } = useQuery({
  queryKey: ["openSessions", partnerId],
  ...
  staleTime: 30_000,
  refetchInterval: effectivePollingInterval,
```

Новый код:
```js
  staleTime: 5_000,
```

### Проверка
- Grep `staleTime` в файле: должна остаться только одна строка с `5_000` в блоке openSessions
- Остальные `staleTime` в файле (если есть) — не трогать

## Fix B: Session-aware grouping в orderGroups (#347)

### Проблема
`orderGroups` группирует все hall-заказы по `tableId`. После закрытия стола и создания новой сессии — заказы старой закрытой сессии и новой открытой сессии оказываются в ОДНОЙ карточке. Это нарушает логику: пользователь видит старые заказы в «Активных» или новые заказы в «Завершённых».

### Wireframe (было → должно быть)
```
БЫЛО:                           ДОЛЖНО БЫТЬ:
Стол 3 [Завершённые]           Стол 3 [Завершённые]    ← старая сессия
  - заказ 1 (closed)             - заказ 1 (closed)
  - заказ 2 (new) ← BUG!
                                Стол 3 [Активные]       ← новая сессия
                                  - заказ 2 (new) ✅
```

### Что менять
Grep: `const orderGroups = useMemo`
Ожидаемое местонахождение: строки ~3768-3819

**Текущая логика (упрощённо):**
```js
visibleOrders.forEach(o => {
  if (o.order_type === 'hall') {
    const tableId = getLinkId(o.table);
    if (!tableGroups[tableId]) {
      tableGroups[tableId] = { type: 'table', id: tableId, orders: [], openSessionId: ... };
      groups.push(tableGroups[tableId]);
    }
    tableGroups[tableId].orders.push(o);
  }
});
```

**Новая логика — группировать по (tableId + sessionId):**
```js
visibleOrders.forEach(o => {
  if (o.order_type === 'hall') {
    const tableId = getLinkId(o.table);
    if (!tableId) return;
    
    const orderSessionId = getLinkId(o.table_session) || 'no-session';
    const openSessionId = openSessionByTableId[tableId]?.id || null;
    
    // Each (table, session) pair = separate group
    const compositeKey = `${tableId}__${orderSessionId}`;
    
    if (!tableGroups[compositeKey]) {
      const tableName = tableMap[tableId]?.name || '?';
      tableGroups[compositeKey] = {
        type: 'table',
        id: tableId,
        sessionId: orderSessionId,         // NEW: track session
        displayName: tableName,
        orders: [],
        openSessionId: openSessionId,       // from openSessionByTableId
      };
      groups.push(tableGroups[compositeKey]);
    }
    tableGroups[compositeKey].orders.push(o);
  }
});
```

**ВАЖНО:** `filteredGroups` использует `group.id` для проверки `openSessionByTableId[group.id]`. После Fix B у нас может быть несколько групп с одним `group.id` (разные сессии). Необходимо дополнительно проверить: группа принадлежит к активной сессии только если `group.sessionId === openSessionByTableId[group.id]?.id`.

Обновить `filteredGroups` (grep: `const hasOpenSession = !!openSessionByTableId`):
```js
// БЫЛО:
const hasOpenSession = !!openSessionByTableId[group.id];

// ДОЛЖНО БЫТЬ:
const isCurrentOpenSession = group.sessionId && 
  openSessionByTableId[group.id]?.id === group.sessionId;
const hasOpenSession = isCurrentOpenSession;
```

Аналогично обновить `tabCounts` (grep: `if (group.type === 'table' && !openSessionByTableId`):
```js
// БЫЛО:
if (group.type === 'table' && !openSessionByTableId[group.id]) {

// ДОЛЖНО БЫТЬ:
const isCurrentOpen = group.type === 'table' && 
  openSessionByTableId[group.id]?.id === group.sessionId;
if (group.type === 'table' && !isCurrentOpen) {
```

Также обновить `activeRequests` binding в orderGroups (строки ~3802-3816): requests привязаны к `tableId`, не к сессии. Запросы о помощи/счёте — прикреплять только к группе с открытой сессией для этого стола:
```js
activeRequests.forEach((req) => {
  const tableId = getLinkId(req.table);
  if (!tableId) return;
  const openSessionId = openSessionByTableId[tableId]?.id;
  // Attach request only to the group matching the open session
  const compositeKey = openSessionId 
    ? `${tableId}__${openSessionId}` 
    : `${tableId}__no-session`;
  if (!tableGroups[compositeKey]) {
    // Create group for request-only card (no orders yet)
    ...
  }
});
```

### Проверка
- `grep -c "compositeKey" staffordersmobile.jsx` → должно быть ≥ 3 вхождений
- `grep "tableGroups\[tableId\]" staffordersmobile.jsx` → должно быть 0 (все заменены на compositeKey)
- После деплоя: закрыть стол → убедиться что скс в «Завершённых». Создать новый заказ → новая скс появляется в «Активных». Старая карточка остаётся в «Завершённых» без изменений.

## Fix C: Invalidate openSessions при появлении новых заказов для стола без сессии

### Проблема
Когда SOM видит новый заказ для стола, у которого `openSessionByTableId[tableId]` = null — это сигнал что в B44 появилась новая TableSession которую SOM ещё не подхватил. Нужно немедленно обновить `openSessions` запрос без ожидания polling.

### Что менять
Grep: `refetchOrders\|invalidateQueries.*orders`
Найти polling useEffect или useEffect который обрабатывает новые заказы (строки ~3700-3770).

Добавить логику: если в `activeOrders` появился заказ для стола без открытой сессии → invalidate `openSessions`:

```js
// Добавить useEffect (после блока activeOrders useMemo, строки ~3617-3640):
useEffect(() => {
  if (!activeOrders?.length) return;
  
  // Check if any hall order belongs to table with no known open session
  const hasOrphanedOrder = activeOrders.some(o => {
    if (o.order_type !== 'hall') return false;
    const tableId = getLinkId(o.table);
    if (!tableId) return false;
    const sessionId = getLinkId(o.table_session);
    if (!sessionId) return false;
    // Has session in B44 but openSessionByTableId doesn't know about it
    return !openSessionByTableId[tableId];
  });
  
  if (hasOrphanedOrder) {
    queryClient.invalidateQueries({ queryKey: ["openSessions"] });
  }
}, [activeOrders, openSessionByTableId, queryClient, getLinkId]);
```

### Проверка
- `grep "hasOrphanedOrder" staffordersmobile.jsx` → 1 вхождение
- Fix должен срабатывать редко (только при несоответствии) — не вызывать лишних запросов
