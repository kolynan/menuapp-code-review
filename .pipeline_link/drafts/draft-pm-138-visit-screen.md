# PM-138: Пустой экран после «Выдан» — UX концепция

## Проблема (текущее поведение)

Когда официант помечает **все позиции заказа** как «Выдан» (статус = served/delivered),
гость в приложении видит **пустой белый экран** в view=orders / CartView.

Нет ни сообщения, ни истории, ни суммы, ни кнопок.

### Почему это плохо
- Гость не понимает: что-то сломалось или заказ выполнен?
- Гость не видит что заказывал и на какую сумму
- Гость не знает как заказать ещё или попросить счёт
- Нет завершённого experience — «обрыв» сессии

### Техконтекст
- Приложение: QR-меню для ресторанов, dine-in (гость за столом, не доставка)
- За один визит гость может сделать **несколько заказов** (закуски → основное → десерт)
- После «Выдан» гость **всё ещё за столом** и может хотеть дозаказать
- Есть help drawer с быстрыми действиями (включая «Принести счёт»)
- Есть система отзывов: гость может оценить блюда после получения

---

## Ключевые потребности гостя после получения заказа

1. **Убедиться** что заказ выполнен (не ошибка)
2. **Видеть** что заказывал + общую сумму визита
3. **Дозаказать** если хочет ещё (вернуться в меню)
4. **Попросить счёт** (финальный шаг визита)
5. **Оставить отзыв** на блюда (PM-137 = редактирование в 7 дней)

---

## Рассмотренные варианты

### Вариант 1 — «Спасибо» экран (простой)
Fullscreen сообщение: «Заказ выдан! 🎉 Спасибо за визит» + кнопка «Вернуться в меню».

**Плюсы:** Простая реализация, чёткий сигнал завершения.
**Минусы:** Гость теряет историю и сумму. Нет дозаказа без отдельной кнопки. Не отвечает на вопрос «сколько я потратил».

### Вариант 2 — Оставлять историю с «Выдано» бейджами
Не прятать выданные позиции, показывать со статусом «Выдано» (серый бейдж).

**Плюсы:** Гость видит историю.
**Минусы:** Нет чёткого завершения, нет суммы, нет CTA для дозаказа/счёта. Просто список.

### Вариант 3 — Auto-redirect в меню + тост
Автоматически закрыть cart и вернуть на меню с тостом «Заказ выдан ✅».

**Плюсы:** Быстро.
**Минусы:** Неожиданно, гость теряет контекст. Не инициировал переход сам.

### Вариант 4 — «Визит-экран» (наша рекомендация до GPT)
Комбинированный экран вместо пустоты:

```
┌─────────────────────────────┐
│  ✅ Заказ выдан             │  ← лёгкий баннер (не fullscreen)
├─────────────────────────────┤
│  Стол 11 · Гость #1313      │
│                             │
│  📋 Ваш заказ:              │
│  ─────────────────────────  │
│  Борщ                 890 ₸ │  ← история позиций
│  Котлета по-киевски  1200 ₸ │     статус: Выдано (серый)
│  Кофе                 450 ₸ │
│  ─────────────────────────  │
│  Итого:              2540 ₸ │  ← общий счёт визита
│                             │
│  [Заказать ещё]  [Счёт 🧾]  │  ← sticky кнопки внизу
└─────────────────────────────┘
```

**«Заказать ещё»** → возвращает в меню (корзина пустая, новый заказ)
**«Счёт 🧾»** → открывает help drawer с типом bill (уже есть 🧾 карточка)

**Плюсы:** Отвечает на все 5 потребностей гостя. Паттерн Wolt/Uber Eats post-delivery. Нет сюрпризов.
**Минусы:** H-вес реализации (нужно реструктурировать CartView). ~$15-20 КС.

## GPT консультация (S179) — выводы

**GPT подтвердил Вариант 4** и усилил:

1. **Группировка по заказам** — не плоский список, а «Заказ 1 · 19:42» / «Заказ 2 · 20:15» с подытогами
2. **«Заказать ещё» = PRIMARY CTA** — в dine-in гость часто хочет десерт/напитки, не сразу счёт
3. **Review block inline** — «Как вам блюда?» + quick stars, НЕ fullscreen прерывание
4. **State model: open tab** — граница = оплата (V7), не выдача (V8). «All served» ≠ визит окончен

## Уточнённые решения (S179 продолжение)

### Сценарий: дозаказ после V8

Когда гость нажал «Заказать ещё» → сделал новый заказ, у него появляются **2 группы заказов** (часть выдана, часть в работе).

**CartView в этом сценарии:**
```
┌─────────────────────────────┐
│  🕐 В работе                │
│  Заказ #3  (принят)         │
│  · Чай зелёный     190 ₸    │
├─────────────────────────────┤
│  ✓ Выданные  ∨  (свернуть)  │
│  Заказ #1 + #2   780 ₸      │
├─────────────────────────────┤
│  Итого всего:     970 ₸     │
└─────────────────────────────┘
```

**Ключевые правила:**
- Активные заказы — **всегда сверху**, выданные — свёрнутый блок снизу
- **V8 (Visit Summary) показывается ТОЛЬКО когда ВСЕ заказы (включая новые) стали «Выдан»**
- Итого в CartView всегда включает ВСЕ заказы визита (выданные + в работе)
- «Выданные» блок разворачивается по тапу

---

### Сценарий: просьбы (Help Drawer)

Где гость видит отправленные запросы (салфетки, счёт и т.д.)?

**Решение: история запросов внутри Help Drawer**

```
[Колокольчик → открывает Help Drawer]
───────────────────────────────
Быстрые запросы:
[Салфетки] [Вилка/ложка] [Другое...]
───────────────────────────────
Отправленные:
✓ Салфетки · 14:32
✓ Счёт · 15:10
```

**Почему так (обоснование):**
- Логично: открыл drawer → видишь что уже отправил → не дублируешь
- CartView не засоряется вспомогательными запросами
- Visit Summary достаточно показывает итог + кнопку «Принести счёт»
- Статус не показываем (staff не отмечает «готово») — только timestamp отправки

**Что НЕ делаем:**
- НЕ показываем запросы в CartView (засорит основной view)
- НЕ показываем запросы в Visit Summary (там только финансовый итог)
- НЕ показываем статус «выполнено/не выполнено» (усложняет workflow персонала)
