## Issues Found

1. **[LOW] Fix 1 Part B — `isCancelledOrder` is not a stable dependency reference**
   `isCancelledOrder` is declared as a regular `const` arrow function (not `useCallback`/`useMemo`) — it is recreated every render. The 3 useMemos in Part B use `isCancelledOrder` inside their callbacks but list `getOrderStatus` in deps (not `isCancelledOrder` itself). This is **correct** — since `isCancelledOrder` closes over `getOrderStatus`, listing `getOrderStatus` in deps ensures the memo recomputes when the predicate would change. However, the prompt could be clearer about WHY `getOrderStatus` is in deps instead of `isCancelledOrder` — an executor unfamiliar with this pattern might add `isCancelledOrder` to the deps array, triggering infinite re-renders.
   **PROMPT FIX:** Add a one-line comment to the deps array: `// getOrderStatus in deps because isCancelledOrder closes over it (not memoized itself)`. No code change needed — just documentation clarity.

2. **[LOW] Fix 3 Step 3.2 — `showTerminal` dependency on `currentTableKey` is implicit**
   `showTerminal = tableIsClosed && !!currentTableKey && !terminalDismissed` — this derived const works correctly because it recalculates on every render. But if an executor tried to "optimize" by wrapping it in a `useMemo`, they might miss `currentTableKey` and `terminalDismissed` as deps. The prompt already has a reviewer note saying "обычные derived values (не hooks)" which mitigates this risk. Minimal concern.
   **PROMPT FIX:** None needed — current documentation is sufficient.

3. **[LOW] Fix 2 Step 2.6 — `getSafeStatus(stageInfo)` vs `getSafeStatus(getOrderStatus(order))`**
   The prompt correctly refactors the status derivation to separate `stageInfo` and `status` variables. The comment explains the guard alignment with `statusBuckets`. However, the original code at lines 880-901 uses `getSafeStatus(getOrderStatus(order))` — a one-line form. The replacement introduces `const stageInfo = getOrderStatus(order)` as a separate line. This is functionally identical and correct, but the executor needs to match the EXACT original code block for the Edit replacement. If the original code diverged even slightly (whitespace, trailing comma), the Edit would fail. The prompt provides the full current code block for matching — this mitigates the risk.
   **PROMPT FIX:** None needed — current code snapshot is adequate for matching.

4. **[LOW] Fix 4 Step 4.1 — `selfTotal === 0` guard vs empty `activeMyOrdersInSession`**
   The guard `if (selfTotal === 0) return null` covers the case where all self-orders are cancelled. But there's a subtle edge case: if all orders have `total_amount = 0` (e.g., complimentary items) AND are active, `selfTotal = 0` → self-block hidden. This is likely acceptable (no money to show), but the prompt doesn't acknowledge this edge case.
   **PROMPT FIX:** Add a brief note: "Edge case: if all active self-orders have total_amount=0, self-block is hidden — acceptable (nothing to attribute in header)." This is informational, not a blocker.

5. **[LOW] Fix 1 Step 1.2 — `pluralizeRu` used for both guest and dish counts**
   The prompt uses `pluralizeRu` with `tr('cart.header.guest_one/few/many')` and `tr('cart.header.dish_one/few/many')`. `pluralizeRu` is verified at line 299. The prompt assumes it follows standard Russian pluralization rules (1=one, 2-4=few, 5+=many). This is standard and correct. No issue.
   **PROMPT FIX:** None.

6. **[LOW] Fix 3 Step 3.4 — `typeof onClose === 'function'` check**
   The terminal onClick uses `typeof onClose === 'function'` with fallback `setView('menu')`. Prop `onClose` is at line 72, `setView` at line 22. Both are props — the `typeof` check is defensive but appropriate since either could theoretically be undefined depending on the calling context. This is fine.
   **PROMPT FIX:** None.
