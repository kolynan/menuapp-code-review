## Fix 1 — Call-site close-table: advance stage_id + close requests

**Status:** `[BUG at line 4143-4155]` + `[NEW CODE]` (добавляем stage advance + request close внутри existing `confirmCloseTable`) + `[MODIFY state shape]` (`closeTableConfirm` @2948 + `handleCloseTable` @4136-4140).

**Target file:** `pages/StaffOrdersMobile/260414-02 StaffOrdersMobile RELEASE.jsx`

**Проблема:** `confirmCloseTable` @4143-4155 зовёт `closeSession(sessionId)`, который обновляет `TableSession.status='closed'` + `Order.status='closed'`, но **НЕ трогает** `Order.stage_id` и `ServiceRequest`. Orders остаются в middle-stage → `hasActiveOrder=true` в filter → стол застревает в Active (если не исчезнет через Fix 2).

**Что делать:**

1. **Расширить `closeTableConfirm` state** @2948:
   ```js
   // БЫЛО: const [closeTableConfirm, setCloseTableConfirm] = useState(null); // { sessionId, tableName } | null
   // СТАЛО: const [closeTableConfirm, setCloseTableConfirm] = useState(null); // { sessionId, tableId, tableName } | null
   ```

2. **Обновить `handleCloseTable`** @4136-4140 — теперь принимает `tableId`:
   ```js
   const handleCloseTable = (tableSessionField, tableName, tableId) => {
     const sessionId = getLinkId(tableSessionField);
     if (!sessionId) return;
     setCloseTableConfirm({ sessionId, tableId, tableName: tableName || 'стол' });
   };
   ```

3. **Найти call-sites `handleCloseTable(` через grep** — их 3 (close-table buttons на @767, @1384, @2381 per ССП finding). В каждом добавить 3-й аргумент `group.id` (tableId). Подтвердить grep'ом что `group.id` доступна в scope каждого вызова.

4. **Расширить `confirmCloseTable`** @4143-4155 — до `await closeSession(sessionId)` добавить stage advance + request close:
   ```js
   const confirmCloseTable = async () => {
     if (!closeTableConfirm) return;
     const { sessionId, tableId } = closeTableConfirm;
     setCloseTableConfirm(null);
     try {
       const finishStage = sortedStages?.find(s => s.internal_code === 'finish');
       if (!finishStage) {
         showToast('Нет этапа "Завершён"');
         return;
       }
       // 1. Advance stage_id for all orders of this session to finish
       const sessionOrders = (orders || []).filter(o =>
         getLinkId(o.table_session) === sessionId &&
         o.stage_id !== finishStage.id &&
         o.status !== 'cancelled'
       );
       if (sessionOrders.length > 0) {
         await runBatchSequential(sessionOrders, (o) =>
           base44.entities.Order.update(o.id, { stage_id: finishStage.id })
         );
       }
       // 2. Close live ServiceRequests for this table
       const liveRequests = (activeRequests || []).filter(r =>
         getLinkId(r.table) === tableId &&
         r.status !== 'done' &&
         r.status !== 'cancelled'
       );
       if (liveRequests.length > 0) {
         await runBatchSequential(liveRequests, (r) =>
           base44.entities.ServiceRequest.update(r.id, { status: 'done' })
         );
       }
       // 3. Close session (TableSession + Order.status bulk)
       await closeSession(sessionId);
       queryClient.invalidateQueries({ queryKey: ["orders"] });
       queryClient.invalidateQueries({ queryKey: ["serviceRequests"] });
       showToast("Стол закрыт");
       setExpandedGroupId(null);
       refetchOrders();
     } catch (err) {
       if (isRateLimitError(err)) {
         queryClient.cancelQueries();
         setRateLimitHit(true);
       }
       showToast("Ошибка при закрытии");
     }
   };
   ```

**НЕ должно быть в Fix 1:**
- НЕ трогать `closeSession()` signature (остаётся `closeSession(sessionId)`) — Fix 3 меняет только тело.
- НЕ менять UI dialogs / toasts / labels («Стол закрыт», «Ошибка при закрытии»).
- НЕ добавлять `__batch: true` к `updateStatusMutation` / `updateRequestMutation` (это Б1b, отдельная КС).
- НЕ трогать `computeTableStatus()` (@1129-1162) — out of scope.
- НЕ менять `handleCloseAllOrders` @4158-4179 — оставляем как reference pattern.

**Связь с Fix 2:** после этого Fix orders получают `status='closed', stage_id=finish`. Но текущий `activeOrders` pre-filter (@3578-3581) **выбросит** их из orderGroups → стол вообще не появится в Completed. **Fix 2 обязателен в этом же КС** иначе Fix 1 не даст визуального эффекта.

**Test cases Fix 1:**
1. Стол с 3 orders в middle stage + 2 ServiceRequest (live) → tap «Закрыть стол» → confirm → все 3 orders: `stage_id=finish, status=closed`, оба request `status=done`.
2. Стол где один order уже в finish stage + один в middle → confirm → finish-order не трогается, middle advanced, оба в итоге `status=closed`.
3. Network failure mid-way → `isRateLimitError` triggers → rate limit banner показывается, стол остаётся в Active для retry.

**Что делать:**

```js
// БЫЛО (@3577-3582):
if (stageId && stagesMap[stageId]) {
  const stage = stagesMap[stageId];

if (stageId && stagesMap[stageId]) {
  const stage = stagesMap[stageId];
```

**НЕ должно быть в Fix 2:**

**Связь с Fix 1:** Fix 2 необходим для того, чтобы orders закрытые через Fix 1 появились в Completed tab. Без Fix 1 orders остаются в middle stage — Fix 2 не навредит, но и не поможет.

## Fix 3 — closeSession Promise.all → sequential

**Status:** `[BUG at line 166-170 components/sessionHelpers.js]`

**Target file:** `components/sessionHelpers.js` (214 строк)

**Проблема:** `Promise.all(sessionOrders.map(Order.update))` @166-170 дёргает N параллельных PATCH — classic 429 trigger на busy tables (10+ orders).

**Что делать:**

Заменить @164-170 на последовательный цикл с delay:
```js
// S267/S282: Bulk-close all non-cancelled orders in this session.
// S282: Sequential with 150ms pacing to avoid B44 429 rate limit.
const sessionOrders = await base44.entities.Order.filter({ table_session: sessionId });
const toClose = sessionOrders.filter(o => o.status !== 'cancelled');
for (const o of toClose) {
  await base44.entities.Order.update(o.id, { status: 'closed' });
  if (toClose.indexOf(o) < toClose.length - 1) {
    await new Promise(resolve => setTimeout(resolve, 150));
  }
}
```

**НЕ должно быть в Fix 3:**
- НЕ менять `closeSession` signature — остаётся `closeSession(sessionId)`.
- НЕ добавлять try/catch внутрь loop — partial failure через throw должен пузыриться в `confirmCloseTable` catch (Fix 1 его ловит).
- НЕ импортировать `runBatchSequential` в sessionHelpers.js — это page-specific helper, не тащим в shared.

**Связь с Fix 1:** Fix 1 уже вызывает `runBatchSequential` для stage_id advance + request close. Fix 3 добавляет sequential и к `closeSession` internal loop. Оба нужны: Fix 1 защищает внешний цикл, Fix 3 — внутренний.

**Test cases Fix 3:**
1. Стол с 12 orders → tap «Закрыть стол» → все 12 закрыты, время ≤ 5 сек, нет 429.
2. Стол с 1 order → нет лишнего delay (последняя итерация skip setTimeout).
3. Network drop mid-close → throw пузырится в Fix 1 `confirmCloseTable` catch → rate limit banner (если 429).
