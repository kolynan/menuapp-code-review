**Recommended Direction**
Use **Option C (no tabs inside the drawer)** for MenuApp’s current UI.  
If you later move this screen to a full page, then use **2 tabs only** (`Мой заказ` / `Стол`).

### Q1: 2 tabs, 3 tabs, or no tabs?
| Option | Cognitive load (non-tech guest) | Practical restaurant value | Implementation complexity | Fit vs references |
|---|---|---|---|---|
| **A: 2 tabs** (`My Order` / `Whole Table`) | Medium | Good | Medium | Good |
| **B: 3 tabs** (`My` / `Others` / `All`) | High | Low-Medium (redundant states) | High | Weak |
| **C: No tabs, improved collapsibles** | **Low (best)** | **High in drawer context** | **Low-Medium (best)** | **Best for this layout** |

Concrete take:
1. **Do not use 3 tabs**. `Others` vs `All` creates mental overhead and duplication.
2. **2 tabs can work**, but cramped at 320px in a drawer and adds mode-switching.
3. **Keep collapsibles**, but make hierarchy clearer:
- `Мой заказ` expanded, with count + subtotal in header.
- `Заказы стола` collapsed, with preview (`3 гостя • 7 позиций • 18 500 ₸`).
- `Итого по столу` always visible.

### Q2: Tab naming (if you still choose tabs)
Best short labels for 320px:
1. **`Мой заказ` / `Стол`** (best balance of clarity + length)
2. `Мой заказ` / `Весь стол` (clear, slightly longer)
3. Avoid `Мои блюда` (too narrow meaning; excludes drinks/add-ons)
4. Avoid `Я` / `Стол` (too abstract for non-tech guests)

Practical tip: add counters in labels if possible: `Мой (2)` / `Стол (8)`.

### Q3: Default tab
Choose **A: always default to `My Order`**.

Reason:
- Stable behavior is easier than smart switching for non-tech users.
- Primary job-to-be-done after scan is “my order”, not table auditing.

If empty order:
- Keep `Мой заказ` open but show empty-state CTA: `Посмотреть заказы стола`.

### Q4: Is “Whole Table” needed?
**Yes, but as secondary information, not primary navigation.**

Real-use scenarios where it helps:
1. Prevent duplicate shared items (“already ordered fries?”).
2. Confirm what friends ordered when discussing bill/split.
3. Quick coordination with waiter (“whose dish is missing?”).
4. Group transparency around table total.

Privacy handling:
1. Show other guests grouped by name/initial.
2. Consider hiding per-person prices by default, while still showing table total.

### What similar products do (practical benchmark)
1. **Wolt Group Order**: shared basket, participants track, host/participant split logic.
2. **Toast**: strong seat/check split model; each guest can pay their own itemized check.
3. **Uber group ordering**: supports pay-self and group tracking.
4. **Starbucks / McDonald’s kiosk / Shake Shack app/kiosk** are mostly personal-cart-first flows.

Inference note: McDonald’s/Shake Shack public materials emphasize single-session order flows and kiosk/app ordering, not multi-guest table-tab UX.
