# Code Review Report: `pages/Profile/base/profile.jsx`

## Summary

The base Profile file has several regressions against the documented Profile behavior and the current PartnerShell contract. The most important problems are incorrect role/restaurant resolution for invited partner staff, role-label i18n failures, and lost mobile-first fixes for the save action and input touch targets.

### Critical Issues (P0) — Must Fix

None.

### High Priority (P1) — Should Fix

- [P1] Profile bypasses `PartnerShell` context and can show the wrong role/restaurant - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L73) re-fetches `base44.auth.me()` and then resolves the restaurant from `userData.partner`, even though the page is always rendered inside [`PartnerShell`](C:/Dev/menuapp-code-review/pages/PartnerShell/260301-00%20PartnerShell%20RELEASE.jsx#L362), which already computes the effective `currentUser`, `userRole`, `partnerId`, and `partner`. For invited `director` / `managing_director` users whose `User.partner` is empty, this page can incorrectly show "not attached" and a mismatched role even while the shell has a valid partner context. FIX: Read `currentUser`, `userRole`, and `partner` from `usePartnerAccess()` inside `ProfileContent`, keep only the editable `fullName` state local, and remove the duplicate `auth.me()` / `Partner.get()` load path.

- [P1] Role badge can be blank or leak raw DB enum values - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L138) returns `tr(\`profile.role.${userRole}\`, userRole)`. If `userRole` is missing, the fallback is `undefined`, so the badge renders with no label. If the translation key is missing, the fallback exposes raw internal values such as `partner_owner` to users. This is a direct i18n/UI regression. FIX: Add an explicit null guard returning `tr("profile.role.unknown", "...")`, and remove the raw-enum fallback so unresolved roles do not surface DB values.

### Medium Priority (P2) — Recommended

- [P2] Hardcoded Russian fallbacks break non-Russian fallback behavior - The local `tr()` helper in [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L41) depends on hardcoded Russian literals passed at every call site, for example [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L126). If an English or Kazakh translation key is missing, the page silently mixes Russian copy into the UI instead of using catalog-driven translations. FIX: Remove language-specific literals from component code and rely on locale keys; if a safety fallback is still needed, use a locale-neutral helper like the current `pages/Profile/profile.jsx` version instead of Russian inline strings.

- [P2] Partner fetch failures are silently shown as "no restaurant" - [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L93) swallows `Partner.get()` errors, and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L206) falls back to `profile.no_restaurant`. That makes "no assigned restaurant" indistinguishable from "restaurant lookup failed", which contradicts the page README and hides real backend failures from users. FIX: Add `isPartnerLoadFailed` state, set it in the partner fetch `catch`, and render a dedicated `profile.restaurant_load_error` message.

- [P2] Save action regressed from a fixed mobile footer to an inline button - The save button stays inside the form card at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L277). On small screens the primary action can fall below the fold again, and the previously documented iOS-safe footer behavior is lost. FIX: Move the save action into a `fixed inset-x-0 bottom-0` footer with safe-area padding and reserve vertical space in the content area.

- [P2] Full Name and Email inputs lost the 44px touch-target fix - The inputs at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L234) and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L244) do not include the documented `min-h-[44px]` mobile sizing. That regresses tap usability on phones. FIX: Add `className="min-h-[44px]"` to both inputs while preserving the email field's read-only styling.

### Low Priority (P3) — Nice to Have

- [P3] `PartnerShell` is rendered without `activeTab="profile"` - The wrapper at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L303) omits the `activeTab` prop, while other Profile variants in this folder pass `activeTab="profile"`. This breaks the established page contract and can prevent correct nav state if Profile is added to shell navigation rules. FIX: Render `<PartnerShell activeTab="profile">`.

- [P3] Nested `min-h-screen` containers create extra viewport height inside `PartnerShell` - The loading, error, and ready roots at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L152), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L163), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L209) each request a full viewport height even though `PartnerShell` already wraps the page in a full-height layout. On mobile this adds unnecessary whitespace and scroll distance. FIX: Replace the inner `min-h-screen` usage with `h-full` / `w-full` or a smaller local min-height that fits the shell content area.

- [P3] Load error state is not announced to assistive tech - The error container at [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L163) replaces the loading UI but has no `role="alert"`. Screen-reader users may not be notified that loading ended with a failure. FIX: Add `role="alert"` to the error-state container and keep the existing readable error text.

- [P3] Decorative icons are exposed to screen readers - Icons such as [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L164), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L171), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L186), [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L194), and [`base/profile.jsx`](C:/Dev/menuapp-code-review/pages/Profile/base/profile.jsx#L218) are purely decorative but lack `aria-hidden="true"`. This adds noise for assistive technologies. FIX: Mark decorative icons as `aria-hidden="true"` and let nearby text provide the accessible label.
