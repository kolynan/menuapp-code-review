## Findings

### [P1] Logout unbind race regressed
- **Lines:** 3261-3273
- **Problem:** `handleLogout()` calls `updateLinkMutation.mutate(...)`, then immediately clears local state and redirects after 500ms. If the network request has not completed, the device can remain bound to the token link. `BUGS.md` already lists this as fixed earlier, so the current file has reintroduced the race.
- **Impact:** Staff can get locked out on the next login because the previous device binding was never reliably cleared.
- **FIX:** Make `handleLogout` async, await `updateLinkMutation.mutateAsync()` or `base44.entities.StaffAccessLink.update(...)`, handle failure with a toast, then clear local storage and navigate only after the unbind attempt finishes.

### [P1] Order status mutations fail silently and do not surface 429 state
- **Lines:** 1594-1606, 1904-1920
- **Problem:** `updateStatusMutation` and `advanceMutation` only roll back optimistic cache in `onError`. They do not show any error toast and do not set `rateLimitHit` on 429 responses, unlike several other mutations in the file.
- **Impact:** Waiters tap an action, the UI snaps back or does nothing, and there is no explanation. In real service flow this looks like a broken button.
- **FIX:** In both `onError` handlers, detect `isRateLimitError(err)` and switch into the rate-limit screen path (`cancelQueries()` + `setRateLimitHit(true)`), and show a translated error toast for all non-429 failures.

### [P1] Partner scoping can fall back to global `.list()` queries
- **Lines:** 3351-3373, 3494-3519
- **Problem:** `tables`, `orders`, and `serviceRequests` use `.list()` when `partnerId` is falsy:
  - `Table.list()` at 3371
  - `Order.list("-created_date", 1000)` at 3496
  - `ServiceRequest.list()` at 3518
  Their `enabled` conditions do not consistently require `partnerId`.
- **Impact:** If link/user data is incomplete or malformed, this page can fetch unscoped cross-restaurant data. In a multi-tenant Base44 app, that is a serious data-isolation bug.
- **FIX:** Require `!!partnerId` before enabling these queries, remove the global `.list()` fallback, and show a blocking error state when partner context cannot be resolved.

### [P1] i18n is still missing across the page
- **Lines:** 220-349, 418-420, 921-924, 2914-2930, 3363, 4139-4186, 4217-4552
- **Problem:** The page uses `useI18n()` only for stage-name translation, but most user-facing UI is still hardcoded in Russian:
  - `STATUS_FLOW`, `TYPE_THEME`, `REQUEST_TYPE_LABELS`, `HALL_UI_TEXT`, `ROLE_LABELS`
  - toast messages and dialog labels
  - header labels and button text
  - fallbacks like `"Ресторан"`, `"стол"`, `"Ссылка недействительна."`
  - `formatRelativeTime()` is hard-wired to `date-fns/locale/ru`
- **Impact:** This violates the project i18n rule and makes the page impossible to localize correctly. `BUGS.md` already tracks this as `BUG-SM-001`, so the issue remains unresolved.
- **FIX:** Convert text constants into functions that accept `t`, replace all user-facing strings with `t('page.section.key')`, and derive date/number locale from the active i18n locale instead of hardcoding Russian formatting.

### [P1] AudioContext cleanup regression
- **Lines:** 1038-1065, 2959-2975
- **Problem:** `createBeep()` allocates an `AudioContext`, and mount logic stores it in `audioRef.current`, but there is no unmount cleanup that calls `ctx.close()`. `BUGS.md` already records this as a previously fixed issue.
- **Impact:** Re-entering this screen repeatedly can leak audio contexts on mobile browsers, eventually breaking sound notifications or hitting browser context limits.
- **FIX:** Add a cleanup effect that calls `audioRef.current?.ctx?.close()` on unmount and then clears the ref.

### [P2] Service request and bulk-close failures still have incomplete error feedback
- **Lines:** 3550-3557, 4167-4178
- **Problem:** `updateRequestMutation` only handles rate-limit failures; all other errors are silent. `handleCloseAllOrders()` also only handles the rate-limit branch and otherwise exits without any feedback.
- **Impact:** Staff can tap request actions or bulk-close flows and receive no visible error when the backend rejects the operation.
- **FIX:** Add translated error toasts for non-429 failures and surface partial batch failures with a summary toast instead of failing silently.

### [P2] Guest-name caching can permanently hide names after transient request failures
- **Lines:** 3617-3653
- **Problem:** Guest IDs are inserted into `loadedGuestIdsRef` before loading starts. Each `SessionGuest.get(id)` failure is swallowed with `.catch(() => null)`, so failed IDs remain marked as attempted and are never retried.
- **Impact:** A temporary network or 429 issue can remove guest badges for the rest of the session, even after connectivity recovers.
- **FIX:** Only mark IDs as loaded after a successful fetch, or remove failed IDs from the ref and retry them with backoff.

### [P2] Double `Стол` prefix regression in legacy cards
- **Lines:** 1530, 1635
- **Problem:** `RequestCard` and `OrderCard` prepend `"Стол "` directly to `tableData.name`. In this codebase, table names are already stored with that prefix, so the UI can render `Стол Стол 5`.
- **Impact:** Reintroduces a previously fixed display regression and makes table identifiers noisy/confusing for staff.
- **FIX:** Reuse the guarded pattern already present at 1881-1885 or extract a shared `withTablePrefix()` helper and use it everywhere table names are rendered.

### [P2] Mobile touch targets are undersized in several high-frequency controls
- **Lines:** 689, 712, 1304, 1558, 2235, 2274, 4231-4251
- **Problem:** Multiple controls are smaller than the 44px mobile-safe target:
  - `RequestCard` action button is `h-7` (28px) at 1558
  - several order/bulk action buttons are `min-h-[36px]`
  - header icon buttons use `w-9 h-9` (36px)
- **Impact:** On a restaurant floor, these are easy to mis-tap, especially one-handed or in motion.
- **FIX:** Raise all interactive controls to at least `44x44`, including icon buttons and small row actions, and expand the favorite-star hit area as well.

### [P2] Production debug logs remain in the page
- **Lines:** 2980-2987
- **Problem:** Session cleanup writes to `console.log()` in production.
- **Impact:** Violates the project rule against debug logs in release code and can leak operational details into production consoles.
- **FIX:** Remove the logs or route them through a guarded debug logger/telemetry hook that is disabled in production.
