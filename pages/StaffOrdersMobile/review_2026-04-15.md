### 1.3 Fix recommendation: A vs B

Choice:
- **B is safer than A** if the team wants to write `Table.status` at all.

- `closeSession()` is currently session-scoped and only receives `sessionId` ([components/sessionHelpers.js:158](components/sessionHelpers.js)).

- B can stay at the page boundary where table context already exists earlier in the flow (`group.id` is the table id in `OrderGroupCard`), so it is easier to reason about and easier to roll back.

### 1.4 Regression risks

1. Multi-entity close can become inconsistent if `TableSession`, `Order`, and `Table` writes are not kept in sync; partial success would desynchronize the UI and backend state.
2. A write-only `Table.status` fix can still leave the `Completed` tab empty, because grouping currently depends on `visibleOrders` and `activeRequests`, not on tables.
3. The legacy fallback close path uses `handleCloseAllOrders()` and only moves orders to finish stage, not `status: "closed"`, so mixed semantics may appear for tables without a valid session id.

### 1.5 Manual test plan

1. Create a hall table with at least one finish-stage/served order and no active service requests; verify the table stays in `Active` before close.
2. Trigger close-table; verify backend state changes together: `TableSession.status = "closed"`, non-cancelled session orders become `status = "closed"`, and the table appears in `Completed` immediately in the waiter UI.
3. Re-test edge cases: table with open requests, table with new/in-progress orders, and legacy/no-session table; confirm close is either blocked correctly or lands in the same completed bucket without disappearing.

### 2.3 Minimal fix pattern (diagnostic only)

Safest minimal pattern:
- Route every UI-triggered mutation, including single-item clicks, through one wrapper that sets a batch marker and suppresses per-mutation invalidation.
- Then issue one explicit invalidate after the write completes.

```jsx
async function mutateOnce(runWrite, invalidateKeys) {
  await runWrite({ __batch: true });
  for (const key of invalidateKeys) {
    queryClient.invalidateQueries({ queryKey: key });
  }
}

// Keep mutation onSuccess/onSettled returning early when vars?.__batch is true.
// Avoid direct base44.entities.*.update(...) in click handlers.
```

Net diagnosis:
- I do **not** see evidence that the main hall single-order path is missing `__batch`.
- I **do** see active non-batched single-click paths elsewhere, plus background `StaffAccessLink` writes that can add rate pressure independently of query invalidation.

## Section 4 - Out-of-scope Risks
