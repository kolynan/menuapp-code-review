### Q1: What should `filteredGroups` / `tabCounts` do when `openId = null`?
**Recommendation:** Use Option A, but narrowly: only force a table group into `completed` when there is a positive mismatch, meaning `openId` exists and `group.sessionId !== openId`. If `openId` is `null`, fall through to the existing order/request status logic.

**Reasoning:** The current bug comes from treating `openId = null` as proof that the group is historical at [staffordersmobile.jsx](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/pages/StaffOrdersMobile/staffordersmobile.jsx:3883). That is too strong. In this file, hall orders without `table_session` are already filtered out earlier at [staffordersmobile.jsx](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/pages/StaffOrdersMobile/staffordersmobile.jsx:3598), so a newly arrived hall order in `orderGroups` normally already has a real `group.sessionId`. When the `orders` query wins the race and `openSessions` is still stale for up to 5 seconds, `openId` being `null` means “session cache is not current yet”, not “this group is completed”. The only authoritative proof that a card is historical is a positive mismatch between the current open session and the group session. Once you keep that distinction, the existing `hasActiveOrder || hasActiveRequest || hasServedButNotClosed` logic correctly keeps new-session orders in `Активные`, while truly inactive tables still land in `Завершённые`.

**Code sketch:**
```js
const openId = openSessionByTableId[group.id]?.id || null;
const isHistoricalGroup = !!openId && group.sessionId !== openId;

if (group.type === 'table' && isHistoricalGroup) {
  return activeTab === 'completed';
}

const hasActiveOrder = group.orders.some((o) => {
  const config = getStatusConfig(o);
  return !config.isFinishStage && o.status !== 'cancelled';
});
const hasActiveRequest = group.type === 'table' &&
  activeRequests.some((r) => getLinkId(r.table) === group.id);
const hasServedButNotClosed = group.orders.some((o) => {
  const config = getStatusConfig(o);
  return config.isFinishStage && o.status !== 'closed' && o.status !== 'cancelled';
});
```

Apply the same predicate in `tabCounts` at [staffordersmobile.jsx](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/pages/StaffOrdersMobile/staffordersmobile.jsx:3908).

**Trade-offs:** For the short stale-cache window, tab placement is derived from order status instead of session ground truth. That is the right trade-off here because it avoids a guaranteed false negative while the cache catches up.

**Mobile UX:** This is the least surprising behavior for waiters. A freshly scanned QR table with a new order stays visible in `Активные` immediately, instead of disappearing into `Завершённые` during the poll race.

**Risks / edge cases:** If a session is marked closed in `TableSession` but its orders are still left non-closed, the fallback status logic could keep it in `Активные`. In the current app contract, `closeSession()` closes all non-cancelled orders after closing the session at [sessionHelpers.js](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/components/sessionHelpers.js:158), so that state should be exceptional rather than normal.

### Q2: How should completed sessions stay separate when `order.table_session` is missing?
**Recommendation:** Use Option D as the primary fix: add a `closedSessions` query and resolve missing historical session IDs by matching each order to a closed `TableSession` for the same table using the session time window. Keep a synthetic fallback key only as a last-resort safety net when no session matches. Do not use timestamp buckets alone as the main grouping rule.

**Reasoning:** The direct merge point is [staffordersmobile.jsx](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/pages/StaffOrdersMobile/staffordersmobile.jsx:3781): `getLinkId(o.table_session) || 'no-session'`. Once multiple historical orders fall into `'no-session'`, they all collapse into one card. A pure fallback like “truncate `created_date` to minute” is not stable enough for restaurant reality: two different seatings can happen close together, and one slow seating can produce orders far apart in time. The only entity that actually knows the historical session boundaries is `TableSession`. The same data also gives you the metadata needed for the completed-card UX in Q3, so one closed-session lookup solves both problems with the smallest conceptual blast radius.

**Code sketch:**
```js
const { data: closedSessions = [] } = useQuery({
  queryKey: ['closedSessions', partnerId],
  queryFn: () => base44.entities.TableSession.filter({ partner: partnerId, status: 'closed' }),
  enabled: canFetch && !!partnerId && !rateLimitHit && !isKitchen,
  staleTime: 30_000,
  retry: shouldRetry,
});

const resolveHistoricalSessionId = useCallback((order) => {
  const linkedId = getLinkId(order.table_session);
  if (linkedId) return linkedId;
  const tableId = getLinkId(order.table);
  const createdTs = safeParseDate(order.created_date).getTime();
  const match = (closedSessionsByTableId[tableId] || []).find((session) => {
    const openedTs = safeParseDate(session.opened_at || session.created_date).getTime();
    const closedTs = safeParseDate(session.closed_at || session.updated_at || session.created_date).getTime();
    return createdTs >= openedTs && createdTs <= closedTs;
  });
  return match?.id || `legacy:${tableId}:${order.id}`;
}, [closedSessionsByTableId]);
```

Then replace the current fallback at [staffordersmobile.jsx](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/pages/StaffOrdersMobile/staffordersmobile.jsx:3781) with `const orderSessionId = resolveHistoricalSessionId(o);`.

**Trade-offs:** This adds one more query plus a small amount of matching logic. The cost is justified because completed-session integrity is not reconstructable from order rows alone once the foreign key is gone.

**Mobile UX:** Separate cards for separate sittings matter more than a tiny bit of extra background work. Waiters need to distinguish “lunch party” from “dinner party” on the same table without opening a merged historical blob.

**Risks / edge cases:** There is one important inconsistency to verify before implementation. Literal `null` `table_session` hall orders are currently filtered out at [staffordersmobile.jsx](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/pages/StaffOrdersMobile/staffordersmobile.jsx:3598), so if B44 truly nullifies the field before `activeOrders` runs, those orders would disappear before grouping and the key fix alone would not surface them. In that case, the grouping fix must be paired with a narrow relaxation of the hall-order filter for finished orders that can be matched to a closed session. If the nullification happens later or comes through a malformed object shape rather than literal `null`, the grouping fix above is sufficient.

### Q3: What should a completed table card show, and where should the data come from?
**Recommendation:** Show a compact “closed session” meta line on the collapsed card, for example `Стол 5 · закрыт 14:32 · 12 400 ₸`. Source the close time from the matched closed `TableSession` and derive the amount from `group.orders`. If there is room, add guest count before the amount. Do not add a new heavy query for dish counts on the collapsed card.

**Reasoning:** The waiter’s question in the completed tab is not “is there work here?” but “which session was this and when did it end?”. The best signal is close time, followed by money. The current codebase already writes `opened_at` and `closed_at` on sessions at [sessionHelpers.js](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/components/sessionHelpers.js:79) and [sessionHelpers.js](/abs/path/C:/Users/ASUS/Dev/Menu%20AI%20Cowork/menuapp-code-review/components/sessionHelpers.js:159). I did not find any established `TableSession.total_amount` usage in this flow, while other pages derive totals from orders, so the safest amount source here is `group.orders.reduce((sum, o) => sum + (Number(o.total_amount) || 0), 0)`. That keeps the UI grounded in data already present in the card and avoids inventing a session field the app does not currently rely on.

**Code sketch:**
```js
const completedSessionMeta = group.type === 'table' &&
  group.sessionMeta &&
  group.sessionId &&
  group.sessionId !== group.openSessionId
  ? {
      closedLabel: formatClockTime(
        group.sessionMeta.closed_at ||
        group.sessionMeta.updated_at ||
        group.sessionMeta.created_date
      ),
      totalLabel: formatHallMoney(
        group.orders.reduce((sum, o) => sum + (Number(o.total_amount) || 0), 0)
      ),
    }
  : null;

{completedSessionMeta && (
  <div className="text-xs text-slate-500">
    {`Закрыт ${completedSessionMeta.closedLabel} · ${completedSessionMeta.totalLabel}`}
  </div>
)}
```

`group.sessionMeta` should come from the same closed-session lookup introduced for Q2, so this does not require a second historical query.

**Trade-offs:** This adds one muted metadata line to completed cards, but it avoids making waiters expand each card just to identify which session ended.

**Mobile UX:** Keep it to one line or one muted chip row. Use time-only for same-day sessions and include the date only when the session is from another day. Avoid fetching `OrderItem` rows just to compute a dish count for the collapsed state; that is not worth the mobile cost.

**Risks / edge cases:** If session metadata is temporarily unavailable, the card should still render using the existing table label plus order-derived total. Close time should degrade gracefully rather than block the card.
