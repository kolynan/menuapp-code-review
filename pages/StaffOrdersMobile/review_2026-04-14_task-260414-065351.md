### High Priority (P1) ‚Äî Should Fix
- [P1] Any `StaffAccessLink` write can incorrectly replace the whole page with `BindingScreen` - `gateView` treats `updateLinkMutation.isPending` as equivalent to device binding (`staffordersmobile.jsx:3262-3265`), but the same mutation object is also used for favorites, cabinet auto-bind, and logout unbind (`3051-3067`, `3072-3086`, `3097-3105`, `3225-3230`). In token mode, a normal favorite toggle can blank the UI behind a fake binding screen. FIX: split device binding into a dedicated mutation, or gate `BindingScreen` only on `tokenState === "bind_needed"` plus a dedicated bind-in-flight flag.

- [P1] Logout unbind race regression is back - `handleLogout` fires `updateLinkMutation.mutate(...)` and immediately clears local state and redirects after 500 ms (`3225-3237`). If the update is slow or fails, the old device can remain bound and the token link stays locked to the current device. `BUGS.md` already records this as a fixed bug. FIX: make logout async, `await` the `StaffAccessLink.update`, handle failure explicitly, and redirect only after the unbind finishes or the user confirms a fallback.

- [P1] Served-orders section is not scoped to the active table session - `servedOrders` is fetched only by `{ table: group.id, status: "served" }` (`1776-1779`). On a reused table, served dishes from a previous session can leak into the current card, which matches the stale-session failure mode documented in the README/BUGS notes. FIX: filter served orders by the current `table_session` (or at minimum by current shift + open session) instead of table ID alone.

- [P1] Ungrouped request cards still bypass the accepted/assignee step - the standalone `RequestCard` action sends `new -> in_progress` directly (`4324`) and never writes `assignee` / `assigned_at`. That contradicts the two-step request flow restored elsewhere in this file and can leave non-table requests without ownership tracking. FIX: align `RequestCard` with the table-request flow: `new/open -> accepted` with assignee metadata, then `accepted -> done`.

- [P1] Restaurant scoping is lost when `partnerId` is missing - non-token users with role `admin` are allowed through `gateView` (`3283-3297`), and the page then falls back to global `.list()` queries for tables, orders, and service requests (`3333-3336`, `3460`, `3482`). That breaks the page contract described in the README and can expose cross-restaurant operational data in a restaurant-specific staff screen. FIX: require a restaurant/partner context for every non-token entry path, or disable all queries when `partnerId` is missing instead of falling back to global lists.

- [P1] The page is still broadly non-compliant with Base44 i18n rules - `useI18n()` is only used for stage-name translation (`2839`), while most user-facing copy is hardcoded in constants and JSX (`213-349`, `2389-2451`, `2475-2581`, `4165-4465`). There are also hardcoded fallbacks such as `partnerData?.name || "–Ý–µ—Å—Ç–æ—Ä–∞–Ω"` (`3327`) and `staffName || "–°–æ—Ç—Ä—É–¥–Ω–∏–∫"` (`2506`). FIX: move all user-visible strings, labels, placeholders, ARIA text, toasts, and fallback copy to `t('page.section.key')`, and convert text arrays/constants into helpers that receive `t`.

### Medium Priority (P2) ‚Äî Recommended
- [P2] Favorites storage mixes table favorites and request favorites, but the "My Tables" UI treats them as one bucket - request favorites are stored with the same `favorites` array as table favorites (`1525-1528`, `3072-3089`), while the My Tables modal and profile sheet count `favorites.length` and `clearAllFavorites()` wipes the entire array (`2389-2390`, `2451`, `2532-2535`, `3097-3108`). Clearing "my tables" also clears starred requests. FIX: separate table and request favorites logically, or derive `tableFavorites` / `requestFavorites` before counting and clearing.

- [P2] Favorite lookup does not normalize IDs on read - writes normalize IDs through `getLinkId()` (`3072-3075`), but `isFavorite` compares the raw `id` directly (`3093-3095`). If Base44 returns numeric IDs or link objects, stars can disappear or fail to match persisted favorites. FIX: normalize `id` inside `isFavorite` the same way as `toggleFavorite`.

- [P2] The double-`–°—Ç–æ–ª` regression is present in multiple live surfaces - the file still hardcodes `–°—Ç–æ–ª ${name}` in request cards (`1512`), single-order cards (`1616-1618`), and the My Tables modal (`2431`). `BUGS.md` already tracks this exact regression family. If table names already include the prefix in the DB, the UI shows `–°—Ç–æ–ª –°—Ç–æ–ª 22`. FIX: reuse one guarded helper such as `withTablePrefix()` / `stripTablePrefix()` everywhere a table label is built.

- [P2] Shift parsing can still return `NaN` and poison the shift filter - `parseTime` in `getShiftStartTime()` just splits and multiplies (`848-851`) and no longer guards `isNaN(h) || isNaN(m)`. That regression can produce invalid shift-start dates from bad `working_hours`, which then corrupts order/request filtering. `BUGS.md` lists this as previously fixed. FIX: restore the `NaN` guard and return `null` for malformed times before computing the shift start.

- [P2] `AudioContext` cleanup regression is back - `createBeep()` allocates an `AudioContext` (`1020-1043`) and mount code stores it in `audioRef` (`2925-2933`), but there is no unmount cleanup that closes `audioRef.current.ctx`. `BUGS.md` already records this leak as fixed once. FIX: add a cleanup `useEffect` that closes the context on unmount and nulls the ref.

- [P2] Most mutations still swallow non-rate-limit failures - order mutations in `OrderCard` and `OrderGroupCard` only roll back cache and never surface an error (`1576-1588`, `1882-1897`); the request mutation only handles rate limits and ignores other failures (`3513-3521`). On a failed update, the optimistic UI can revert with no explanation to the waiter. FIX: add user-visible error feedback for all mutation failures and funnel rate-limit failures into the same handling path.

- [P2] Request assignee checks are not normalized - the requests section uses `request.assignee === effectiveUserId` (`2333`) even though the rest of the file normalizes Base44 link fields with `getLinkId()`. If `request.assignee` is an object/value-link, the assignee pill never appears and "assigned to me" logic is wrong. FIX: compare `getLinkId(request.assignee)` to `effectiveUserId`.

### Low Priority (P3) ‚Äî Nice to Have
- [P3] Production debug logging is still present - the session cleanup effect prints to `console.log` on success and failure (`2943-2951`). Project rules explicitly call out console logging as a production violation. FIX: remove the logs or route them through a controlled debug/logger flag that is off in production.

- [P3] Sort-mode radios can fire the handler twice per tap - each sort option binds `onClick` on the wrapping `<label>` and `onChange` on the nested `<input>` (`2675-2681`, `2698-2704`). A single tap can therefore call `onChangeSortMode()` twice, causing duplicate toasts and duplicate writes. FIX: keep the handler in one place only, preferably the radio `onChange`.
